## 一、用户是谁，痛点是什么

**用户**: 小米 AIOT 部门的 Android 系统性能优化工程师（约 20-30 人）。

**工作流**: 在多台手机上跑自动化测试（内存压力测试、流畅性测试、功耗测试），每次产出一个 ZIP 包，里面包含：
- showmap 文件（内存映射快照，文本格式）
- perf.data 文件（CPU 采样数据，二进制格式）
- perfetto trace 文件（系统 trace，二进制格式）
- 各种 CSV 统计数据

**痛点**:

| 痛点 | 之前怎么做 | 有多痛 |
|------|-----------|--------|
| 数据散落 | 每次测试的 ZIP 包存在各自电脑上 | 找历史数据要问人 |
| 对比困难 | 用 Excel 手动对比两台设备的内存数据 | 一次对比要 2-3 小时 |
| 格式复杂 | showmap/perf.data/trace 各有各的工具 | 切换工具上下文成本高 |
| 无法追溯 | 没有统一的数据管理 | 测试结果不可复现 |
| 协作困难 | 分析结果截图发群里 | 信息丢失，无法二次分析 |

**核心价值主张**: 把"上传 ZIP → 自动解析 → 多维度可视化对比"做成一站式平台，将数小时的人工分析缩短到分钟级。

## 二、核心需求拆解

从用户痛点反推功能需求：

```
痛点: 数据散落 → 需求: 统一上传和管理
  → 功能: ZIP 上传 → 自动解压 → 项目列表 → 侧边栏导航

痛点: 对比困难 → 需求: 自动化多设备对比
  → 功能: DiffTable 通用对比组件 → 自动计算差值和百分比 → 基准设备切换

痛点: 格式复杂 → 需求: 统一可视化
  → 功能: 内存/功耗/流畅性/冷启动四大模块 → 火焰图 → Perfetto 集成

痛点: 无法追溯 → 需求: 数据持久化
  → 功能: Doris 数据库存储 → 历史数据查询 → 项目状态管理

痛点: 协作困难 → 需求: 在线共享
  → 功能: Web 平台 → URL 分享 → 多人同时查看
```

## 三、关键决策与 trade-off

### 3.1 为什么做平台而不是脚本工具

备选方案：写一套 Python 脚本，命令行跑，输出 HTML 报告。

选择做 Web 平台的原因：
- 脚本工具解决不了"协作"和"追溯"的痛点
- 测试工程师不是开发者，不愿意装 Python 环境跑脚本
- Web 平台可以渐进式迭代，先做核心功能再逐步完善
- 数据集中存储后可以做跨项目对比、趋势分析等高级功能

trade-off：开发成本高很多，但长期价值大。

### 3.2 为什么前端做重而不是后端做重

这个项目的前端比一般的管理后台重很多（ECharts 图表、D3 火焰图、Perfetto 集成、IndexedDB）。

原因：性能分析是**交互密集型**场景，用户需要：
- 在图表上点选数据点做对比（不是看静态报表）
- 在火焰图上缩放、搜索、导航（不是看截图）
- 在 Perfetto 里拖拽时间轴分析 trace（不是看文本）

这些交互必须在前端完成，后端只负责数据查询和文件服务。

### 3.3 为什么用消息队列解耦上传和解析

备选方案：上传接口同步解析，解析完再返回。

选择 MQ 异步的原因：
- 一个项目的数据解析可能要 10-30 分钟（GB 级 trace 文件）
- 同步处理会阻塞上传接口，用户体验极差
- MQ 解耦后上传秒级返回，前端轮询解析状态
- Consumer 可以独立部署和扩缩容
- 失败可重试，不影响上传链路

trade-off：引入了 MQ 的运维复杂度，但对于长耗时任务这是标准做法。

### 3.4 为什么选 Doris 而不是 MySQL

数据特点：
- 写入：批量写入（Python Consumer 解析完一次性灌入），不需要事务
- 查询：多维度聚合（按设备、迭代、分类分组求和），不需要复杂 JOIN
- 数据量：单项目几十万行，全量百万级

Doris 的优势：
- 列式存储，聚合查询比 MySQL 快 10-100 倍
- 兼容 MySQL 协议，学习成本低（JdbcTemplate 直接用）
- Stream Load 支持 Parquet 批量写入，和 Python 数据处理管道天然契合
- MPP 架构，数据量增长后可以水平扩展

trade-off：不适合高频小事务写入（但这个场景不需要），生态不如 MySQL 成熟。

### 3.5 为什么自建火焰图而不是直接用 Perfetto

Perfetto 擅长时序 trace 分析，但不擅长 CPU 采样数据的聚合对比。

用户需要的是：
- 按函数/库聚合 CPU 采样计数 → Perfetto 做不了
- 两台设备的火焰图 diff 对比 → Perfetto 做不了
- 在函数列表里搜索、排序、分页 → Perfetto 做不了

所以用 D3 Flame Graph 自建，同时保留 Perfetto 集成用于 trace 分析。两个工具互补，不是替代关系。

### 3.6 大文件上传：从问题到方案的决策过程

**触发**: 用户反馈 11GB 的测试数据上传失败。

**排查**:
1. 浏览器报 CORS 错误 → 误导
2. 深入看 Network 面板 → 413 Request Entity Too Large
3. FDS 单次 PUT 限制 5GB

**方案评估**:

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 提高 FDS 限制 | 零代码改动 | 需要 FDS 运维配合，不一定能改 | ❌ |
| 前端压缩后上传 | 减小体积 | ZIP 已经是压缩格式，压缩率极低 | ❌ |
| 走 BFF 代理上传 | 不涉及 FDS | BFF 内存压力大，11GB 要流式处理 | 可行但复杂 |
| FDS 分片上传 (MPU) | 标准做法，FDS SDK 支持 | 需要前后端改动 | ✅ |

**实施中的发现**:
- FDS 分片上传必须按顺序（和 S3 不同）→ 只能串行
- 预签名 URL 的 Content-Type 必须和请求一致 → 签名校验严格
- 浏览器 PUT 触发 CORS 预检 → 每片实际 2 个请求

**最终方案**: FDS MPU 串行上传，64MB/片，后端 complete 合并 + 通知 Talos。< 4GB 走原有逻辑，向后兼容。

## 四、产品迭代思维

这个项目不是一次性做完的，是渐进式迭代：

```
V1: 冷启动分析（最初需求）
    上传 ZIP → Nuxt BFF 解压 → 本地文件 → ECharts 图表 + DiffTable

V2: 加入流畅性分析
    C++ Native Addon 解析 perf.data → 火焰图

V3: 加入功耗分析
    电流曲线 + 指令数趋势

V4: 数据持久化（从文件到数据库）
    Python Consumer + Doris → 不再依赖本地文件

V5: 内存分析模块（最复杂的模块）
    9 张 Doris 表 + 19 个 API + 多级下钻 + Perfetto 集成

V6: 大文件上传支持
    FDS MPU 分片上传
```

每个版本都是解决一个具体的用户痛点，不是一开始就设计好所有功能。这也是为什么项目里有两套存储（FDS + JuiceFS）、两套上传链路——不是设计失误，是迭代过程中的自然演化。
