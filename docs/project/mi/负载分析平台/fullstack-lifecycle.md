## 全生命周期总览

```
需求分析 → 架构设计 → 项目管理 → 开发实施 → 质量保障 → 部署上线
    ↑                                                        │
    └──── 监控运维 ← 技术债务管理 ← 文档沉淀 ← 迭代反馈 ←──┘
```

一个人做全栈，最大的挑战不是技术，而是**角色切换**。你同时是产品经理、架构师、前端、后端、DBA、运维、测试。关键是建立一套轻量但完整的流程，让每个环节都不掉链子。

## 一、需求分析与立项

### 1.1 识别真实痛点

不要用户说什么就做什么，要挖到根因。


**LoadTerminator 的例子**：

| 用户说的 | 真实痛点 | 解决方案 |
|---------|---------|---------|
| "能不能帮我写个脚本对比 showmap" | 每次手动对比太慢 | 做成 Web 平台，自动解析+可视化 |
| "我想看两台手机的内存差异" | 多设备对比没有统一工具 | DiffTable 通用对比组件 |
| "文件太大传不上去" | FDS 5GB 限制 | 分片上传 MPU |

**方法**：
- 跟用户坐在一起看他怎么工作（不是听他描述）
- 记录他重复做的事情 → 这就是自动化的机会
- 问"如果这个问题解决了，你一天能省多少时间" → 量化价值

### 1.2 需求优先级排序

用 **影响面 × 频率 × 痛苦程度** 三维评估：

```
P0 (必须做): 高影响 × 高频 × 高痛苦
   例: 上传解析 → 每天都用，没有就没法工作

P1 (应该做): 高影响 × 中频
   例: 多设备对比 → 每周几次，手动做要 2 小时

P2 (可以做): 低影响 or 低频
   例: 导出 Excel → 偶尔需要，不急

P3 (先不做): 锦上添花
   例: 国际化 → 内部工具，全是中国人用
```

### 1.3 MVP 定义

MVP 不是"功能最少"，是"能跑通核心链路的最小集合"。

**LoadTerminator V1 的 MVP**：
```
上传 ZIP → 解压 → 解析 showmap → 折线图展示 → 两台设备对比
```

砍掉的：火焰图、Perfetto 集成、功耗分析、权限管理、Docker 部署。这些都是后续迭代加的。

**MVP 检验标准**：给用户用一天，他愿不愿意继续用。如果愿意，说明核心价值成立。

## 二、架构设计与技术选型

### 2.1 架构设计三原则

1. **务实优先**：一个人做，不追求完美架构，追求能跑、能维护、能迭代
2. **每层用最合适的工具**：不强求统一技术栈（C++ 做解析、Python 做数据处理、Java 做 API、TypeScript 做前端）
3. **渐进式演化**：先解决当前问题，预留扩展点，不提前过度设计

**反面教材**：一开始就设计微服务、消息队列、分布式缓存 → 内部工具起步阶段，单体就够了。

### 2.2 技术选型决策框架

每个选型决策记录三个问题：

| 问题 | 说明 |
|------|------|
| 为什么选它 | 解决什么具体问题 |
| 为什么不选替代方案 | 替代方案的缺点是什么 |
| 什么时候需要换 | 到什么规模/场景需要迁移 |

**示例 — 为什么选 Doris 不选 MySQL**：

```
为什么选: 列式存储，聚合查询快 10-100x，兼容 MySQL 协议
为什么不选 MySQL: 百万行多维度 GROUP BY 太慢
什么时候需要换: 如果需要高频事务写入（当前场景不需要）
```

### 2.3 数据库选型实战

| 场景 | 选择 | 理由 |
|------|------|------|
| 分析数据（内存/功耗） | Doris (OLAP) | 列式存储，聚合查询快，Stream Load 批量写入 |
| 用户权限 | lowdb (JSON 文件) | 20 个用户，不值得上数据库 |
| 缓存/协调 | Redis | 分布式锁 + Pub/Sub，标准选择 |
| 大文件 | FDS 对象存储 | 公司基础设施，不用自己运维 |
| 前端大数据 | IndexedDB | 浏览器端，几十 MB 数据分页查询 |

**原则**：不同数据用不同存储，不要把所有东西塞进一个 MySQL。

## 三、项目管理与迭代节奏

### 3.1 一个人的迭代管理

一个人做全栈，最容易犯的错是"想到什么做什么"，没有节奏。

**推荐节奏**：2 周一个迭代

```
第 1 天:   规划（确定这个迭代做什么，拆任务）
第 2-8 天: 开发（按优先级逐个完成）
第 9 天:   自测 + 修 bug
第 10 天:  部署 + 给用户用 + 收集反馈
```

**关键习惯**：
- 每个迭代开始前写一句话目标："这个迭代结束后，用户能做什么之前做不了的事"
- 每天结束前花 5 分钟记录：今天做了什么、卡在哪里、明天做什么
- 不要在迭代中途加需求，记到下个迭代

### 3.2 任务拆解方法

一个功能从需求到上线，拆成这些任务：

```
以"内存分析模块"为例：

1. [设计] 数据库表结构设计 (0.5天)
2. [后端] Doris 建表 + 数据导入验证 (0.5天)
3. [后端] 19 个 API 接口实现 (3天)
4. [后端] 接口自测 (curl 验证) (0.5天)
5. [前端] 页面路由 + 布局 (0.5天)
6. [前端] 多轮峰值折线图 (1天)
7. [前端] 单轮峰值图 (1天)
8. [前端] DiffTable 对比表格 (2天)
9. [联调] 前后端联调 + 修数据格式问题 (1天)
10. [部署] 部署到 staging + 用户验收 (0.5天)
```

**拆解原则**：
- 每个任务不超过 1 天
- 每个任务有明确的完成标准（"curl 返回 200 且数据正确"）
- 前后端任务可以并行（先定好接口契约）

### 3.3 版本规划实战

LoadTerminator 的实际版本演进：

```
V1 (2周): 冷启动分析 — MVP
   目标: 上传 ZIP → 解析 → 折线图 + 对比表
   价值: 用户开始用了，验证核心链路

V2 (2周): 流畅性分析
   目标: C++ Addon 解析 perf.data → 火焰图
   价值: 覆盖第二个分析维度

V3 (1周): 功耗分析
   目标: 电流曲线 + 指令数趋势
   价值: 覆盖第三个维度

V4 (3周): 数据持久化
   目标: Python Consumer + Doris，数据不再丢失
   价值: 从"工具"变成"平台"

V5 (4周): 内存分析（最复杂）
   目标: 9 张表 + 19 个 API + 多级下钻
   价值: 覆盖用户最痛的分析场景

V6 (1周): 大文件上传
   目标: FDS MPU 分片上传
   价值: 解决 11GB 文件传不上去的问题
```

**规律**：每个版本解决一个具体问题，不贪多。用户反馈驱动下一个版本的方向。

### 3.4 与其他角色的协作

一个人做全栈不代表完全独立，你需要和这些人协作：

| 角色 | 协作内容 | 沟通频率 |
|------|---------|---------|
| 测试工程师（用户） | 需求收集、验收反馈 | 每周 1-2 次 |
| 后端同事 | Java 后端存量代码维护 | 按需 |
| 运维/SRE | 容器平台部署、域名配置 | 部署时 |
| 数据工程师 | Doris 集群运维、表结构变更 | 按需 |
| 主管 | 进度同步、资源协调 | 每周 1 次 |

**沟通技巧**：
- 给非技术人员看效果（截图/录屏），不要讲技术细节
- 给技术同事看接口文档和数据格式，不要口头描述
- 进度同步用"完成了什么 + 下一步做什么 + 有什么风险"三段式

## 四、开发实施

### 4.1 全栈开发顺序

推荐顺序：**数据库 → 后端 API → 前端页面 → 联调 → 部署**

```
为什么这个顺序？

1. 数据库先行: 表结构决定了数据模型，改表比改代码成本高
2. 后端先行: API 是前后端的契约，先定好接口前端才能开发
3. 前端后做: 可以用 mock 数据先开发，联调时再接真实 API
4. 联调收尾: 数据格式对齐、边界情况处理
5. 部署最后: 功能完整后再部署，避免半成品上线
```

**例外情况**：如果是纯前端功能（如火焰图渲染），可以前端先行。

### 4.2 前后端联调策略

```
开发阶段:
  前端 → devProxy → 线上 staging 后端（大部分接口）
  前端 → localhost:8090（正在开发的新接口）

联调阶段:
  前端 → devProxy → localhost:8090（全部走本地后端）
  curl 先验证后端接口 → 再接前端

上线阶段:
  前端 → nginx → 线上后端
```

**联调 checklist**：
- [ ] 接口路径一致（前端调的和后端注册的一样）
- [ ] 请求方法一致（GET vs POST）
- [ ] 参数名一致（project vs projectId）
- [ ] 响应格式一致（entity vs data）
- [ ] 空数据处理（接口返回空数组时前端不崩）
- [ ] 错误处理（接口 500 时前端有提示）

### 4.3 代码组织原则

**前端**：
```
pages/        → 路由页面，只做布局和组合
components/   → UI 组件，三层分离（基础/公共业务/业务）
composables/  → 可复用逻辑（provide/inject, hooks）
services/     → API 调用，统一管理接口地址
stores/       → 全局状态，只放真正全局的数据
```

**后端**：
```
controller/   → 接收请求，参数校验，不写业务逻辑
service/      → 业务逻辑，PO → VO 转换
infra/doris/  → 数据访问，SQL 拼接
model/        → 数据模型（PO/VO/Req 分开）
config/       → 配置类（数据源/认证/CORS）
```

**原则**：每层只做自己的事。Controller 不写 SQL，Service 不处理 HTTP，DorisService 不做业务聚合。

## 五、质量保障与测试策略

### 5.1 内部工具的测试策略

内部工具（上百人的部门在用）不需要和 ToC 产品一样的测试覆盖率。务实的策略：

| 层级 | 策略 | 投入 |
|------|------|------|
| 后端 API | curl/Postman 手动测试每个接口 | 必须做 |
| 前端页面 | 人肉点击测试核心流程 | 必须做 |
| 数据一致性 | SQL 查询验证数据正确性 | 必须做 |
| 单元测试 | 核心工具函数写测试 | 选做 |
| E2E 测试 | 不做 | 投入产出比太低 |

**核心原则**：把时间花在最容易出问题的地方 — 数据正确性和接口联调。

### 5.2 接口测试实战

每个后端接口上线前，用 curl 验证：

```bash
# 1. 正常请求
curl "http://localhost:8090/dfp-api/memory/iters-peak-rss-by-category?project=xxx"
# 验证: code=200, entity 有数据, 字段名正确

# 2. 空数据
curl "http://localhost:8090/dfp-api/memory/iters-peak-rss-by-category?project=不存在的项目"
# 验证: code=200, entity=[], 不报错

# 3. 缺少参数
curl "http://localhost:8090/dfp-api/memory/iters-peak-rss-by-category"
# 验证: 返回 400 或有意义的错误信息

# 4. POST 接口带 body
curl -X POST "http://localhost:8090/dfp-api/memory/round-object-diff" \
  -H "Content-Type: application/json" \
  -d '{"project":"xxx","devices":[{"device":"o1","iter":1}]}'
```

**记录测试结果**：每个接口的测试结果记到文档里（本项目的 `memory_api_test_report.md`），方便回归。

### 5.3 前端质量保障

```
1. ESLint + Prettier: 代码格式统一，CI 里强制检查
2. TypeScript: 类型安全，接口返回值定义类型
3. 核心流程手动测试:
   - 上传 → 解析 → 查看结果（完整链路）
   - 切换 RSS/PSS → 图表和表格都响应
   - 空数据 → 不白屏，有提示
   - 浏览器刷新 → 状态恢复（Pinia persistedstate）
```

### 5.4 数据一致性校验

数据从 Python Consumer 写入 Doris，到 Java API 查出来，到前端展示，每一步都可能出错。

```sql
-- 验证数据写入正确
SELECT COUNT(*) FROM mem_iter_peak_rss_category WHERE project = 'xxx';
-- 预期: 和 Python 解析的行数一致

-- 验证聚合逻辑正确
SELECT iter, device, SUM(rss) as total_rss
FROM mem_iter_peak_rss_category
WHERE project = 'xxx'
GROUP BY iter, device;
-- 预期: 和 API 返回的数据一致

-- 验证前端展示正确
-- 打开浏览器 DevTools → Network → 看接口返回值
-- 和页面上显示的数字对比
```

## 六、CI/CD 与部署

### 6.1 部署流水线设计

```
git push → GitLab CI 触发 → Kaniko 构建镜像 → 推送 micr → 容器平台拉取 → 滚动更新
```

**关键配置**：
- `--frozen-lockfile`：保证 CI 和本地依赖一致
- `NODE_OPTIONS=--max-old-space-size=4096`：防止构建 OOM
- base 镜像分层：编译工具链不变时复用缓存，CI 从 20 分钟降到 5 分钟

### 6.2 多环境管理

| 环境 | 用途 | 部署方式 | 数据 |
|------|------|---------|------|
| dev | 本地开发 | `pnpm dev` | devProxy 到 staging |
| staging | 测试验证 | CI 自动部署 | staging Doris |
| production | 线上使用 | 手动触发部署 | production Doris |

**原则**：staging 和 production 用同一个 Dockerfile，只是环境变量不同。代码在 staging 验证通过后才部署 production。

### 6.3 回滚策略

```
发现线上问题:
  1. 容器平台回滚到上一个镜像版本（1 分钟）
  2. 排查问题，修复后重新部署

数据库变更回滚:
  Doris 加列是向后兼容的（老代码不查新列就不受影响）
  删列/改列名要谨慎，先确认没有代码依赖再操作
```

## 七、监控运维与线上问题处理

### 7.1 内部工具的监控策略

不需要 Prometheus + Grafana 全套，但这些必须有：

| 监控项 | 方式 | 目的 |
|--------|------|------|
| 应用是否存活 | 容器平台健康检查 | 挂了自动重启 |
| 接口是否正常 | 用户反馈 + 定期手动检查 | 发现问题 |
| 错误日志 | 容器平台日志收集 | 排查问题 |
| 磁盘空间 | trace-fs 挂载点监控 | 防止磁盘满 |

**如果要进阶**：
- 接口响应时间监控（Spring Boot Actuator）
- 前端错误上报（window.onerror → 发到后端）
- Doris 慢查询日志

### 7.2 日志设计

**后端日志规范**：
```java
// 好的日志: 有上下文，能定位问题
log.info("查询内存峰值, project: {}, 返回 {} 条", project, result.size());
log.error("查询失败, project: {}, error: {}", project, e.getMessage(), e);

// 差的日志: 没有上下文
log.info("查询成功");
log.error("出错了");
```

**前端日志**：
```typescript
// 关键操作打 console.log（开发时有用，上线前清理或用 debug 级别）
console.log('[memory] API response:', { project, rows: rows.length });

// 错误统一处理
catch (error) {
  console.error('[memory] 查询失败:', error);
  message.error('查询失败，请稍后重试');
}
```

### 7.3 线上问题排查流程

```
1. 确认现象: 用户报什么问题？能复现吗？
2. 看前端: 浏览器 DevTools → Network → 哪个请求失败了？状态码是什么？
3. 看后端: 容器日志 → 有没有报错？SQL 执行了吗？
4. 看数据: Doris 里有没有数据？数据对不对？
5. 定位根因: 是前端问题、后端问题、数据问题、还是网络问题？
6. 修复 + 验证 + 部署
```

### 7.4 常见线上问题与应对

| 问题 | 现象 | 排查方向 | 解决方案 |
|------|------|---------|---------|
| 页面白屏 | 打开页面什么都没有 | Console 报错 → JS 错误 | 修复代码，加错误边界 |
| 接口 401 | 请求被拒绝 | Aegis 认证失败 | 检查 IGNORE_URL 配置 |
| 接口 404 | 找不到接口 | 路径不匹配 | 检查 Controller 路径和代理配置 |
| 接口 500 | 服务端报错 | 看后端日志 | 修复 Java 代码 |
| 数据不对 | 页面显示的数字和预期不符 | SQL 查询验证 | 修复 SQL 或数据导入 |
| 上传失败 | 文件传不上去 | 看文件大小、网络、FDS 状态 | 检查 MPU 逻辑 |
| 页面卡顿 | 操作响应慢 | Performance 面板 | 优化渲染/减少数据量 |

## 八、技术债务管理

### 8.1 识别技术债务

技术债务不是"代码写得烂"，是"为了快速交付做的妥协，未来需要还的"。

**常见来源**：
- 为了赶工跳过的重构
- 临时的 hack（硬编码、TODO 注释）
- 过时的依赖版本
- 缺失的错误处理
- 不一致的命名/风格

### 8.2 债务分级与处理策略

| 级别 | 定义 | 处理时机 |
|------|------|---------|
| 高危 | 影响线上稳定性或数据正确性 | 立即修 |
| 中等 | 影响开发效率或可维护性 | 下个迭代安排 |
| 低等 | 代码风格、命名不一致 | 顺手改，不专门安排 |
| 可接受 | 当前方案够用，未来可能需要换 | 记录，到时候再说 |

### 8.3 本项目的技术债务清单

| 债务 | 级别 | 状态 | 计划 |
|------|------|------|------|
| JdbcTemplate deprecated API | 低等 | 已知 | 升级 Spring Boot 时一起改 |
| FDS + S3 双存储并存 | 可接受 | 已知 | 等公司统一存储方案 |
| 部分接口缺少分页 | 中等 | 已知 | 数据量大的接口优先加 |
| console.log 调试代码残留 | 低等 | 持续清理 | 提交前检查 |
| 前端缺少错误边界组件 | 中等 | 待做 | 下个迭代 |
| Redis 缓存策略未实现 | 中等 | 待做 | 查询慢的接口优先加 |

**管理方式**：维护一个简单的 markdown 清单，每个迭代开始时 review 一遍，挑 1-2 个中等级别的顺手还掉。

## 九、文档与知识沉淀

### 9.1 必须写的文档

| 文档 | 内容 | 给谁看 |
|------|------|--------|
| README | 项目介绍、如何启动、如何部署 | 新人/接手者 |
| API 文档 | 接口路径、参数、返回值 | 前端开发 |
| 数据库设计 | 表结构、字段说明、索引 | 后端开发/DBA |
| 架构文档 | 系统架构图、技术选型理由 | 架构评审/交接 |
| 部署文档 | CI/CD 配置、环境变量、运维操作 | 运维/自己 |
| 变更日志 | 每个版本做了什么 | 用户/主管 |

### 9.2 文档维护策略

**写文档的时机**：
- 做完一个功能模块后，趁热写
- 踩了一个坑后，立刻记录
- 不要攒到最后一起写（你会忘掉细节）

**文档放在哪**：
- 和代码放一起（`docs/` 目录），跟着代码版本走
- 不要放在 Wiki 或 Confluence 里（和代码脱节，容易过时）

**保持文档不过时**：
- 改了接口就更新 API 文档（养成习惯）
- 改了表结构就更新数据库文档
- 如果文档和代码不一致，以代码为准，然后更新文档

## 十、项目交接与可持续性

### 10.1 交接清单

如果你要把项目交给别人，确保这些东西都有：

```
□ 代码仓库权限
□ README（如何启动、如何部署）
□ 环境变量清单（哪些需要配置、怎么获取）
□ 数据库访问权限 + 表结构文档
□ CI/CD 配置说明
□ 容器平台访问权限
□ 域名和证书信息
□ 第三方服务账号（FDS、Talos、Aegis）
□ 已知问题和技术债务清单
□ 关键联系人（运维、DBA、用户代表）
```

### 10.2 让项目能活下去

一个人做的项目，最大的风险是"人走了项目就死了"。

**降低 bus factor 的方法**：
- 代码规范统一（ESLint + Prettier），别人能读懂
- 关键决策写文档（为什么这么做，不只是怎么做）
- 不要有只存在你脑子里的知识（密码、配置、操作步骤都写下来）
- CI/CD 自动化（不要依赖手动操作）
- 定期给同事 demo（让至少一个人了解项目全貌）

```

  □ 空数据/错误情况处理

部署阶段:
  □ staging 环境验证通过
  □ 用户验收通过
  □ 部署到 production

  □ 文档更新
  □ 技术债务记录
```
