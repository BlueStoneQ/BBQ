## 一、架构决策原则

这个项目的架构决策遵循三个原则：

1. **务实优先**: 一个人做全栈，不追求完美架构，追求能跑、能维护、能迭代
2. **每层用最合适的语言**: 不强求统一技术栈，C++ 做解析、Python 做数据处理、Java 做 API、TypeScript 做前端
3. **渐进式演化**: 不一开始就设计终态，先解决当前问题，后续再重构

## 二、技术选型与理由

### 2.1 为什么 Nuxt 而不是纯 Vue

| 需求 | 纯 Vue (Vite) | Nuxt |
|------|--------------|------|
| SPA 前端 | ✅ | ✅ |
| 服务端 API (BFF) | ❌ 需要另起 Express | ✅ Nitro 内置 |
| 文件系统路由 | ❌ 手动配置 | ✅ 自动生成 |
| 服务端中间件 | ❌ | ✅ auth/gzip/role |
| PM2 部署 | 需要额外配置 | ✅ `.output/server/index.mjs` 直接跑 |

核心原因：项目需要 BFF 层（文件上传/解压、C++ Addon 调用、Redis 协调），Nuxt 的 Nitro Server 让前端和 BFF 在同一个项目里，不需要维护两个仓库。

### 2.2 为什么需要 BFF 层

```
没有 BFF:
浏览器 → Java 后端（所有事都做）
问题: Java 后端要处理文件上传/解压、调用 C++ 库、管理 Redis，职责太重

有 BFF:
浏览器 → Nuxt BFF（文件操作、C++ 解析、Redis 协调）
浏览器 → Java 后端（数据库查询、文件服务、认证）
```

BFF 承担的是"前端无法直接做，但又不属于业务后端"的事：
- 文件上传/解压：需要文件系统操作
- simpleperf 解析：需要 C++ Native Addon
- Redis 分布式锁：需要服务端网络访问
- FDS 预加载：需要服务端下载大文件

### 2.3 为什么 4 种语言

| 语言 | 用在哪 | 为什么不用其他 |
|------|--------|--------------|
| TypeScript | 前端 + BFF | 前端天然选择，BFF 和前端同仓库 |
| C++ | simpleperf 解析 | 官方只提供 C 库 `libsimpleperf_report.so`，没有 JS/Java 绑定 |
| Java | 业务后端 | 公司技术栈，团队其他人维护的存量代码 |
| Python | 数据处理 | showmap/ebpf 解析脚本是 Python 生态（perfetto SDK、pyelftools） |

不是为了炫技用 4 种语言，是每层的生态决定了语言选择。如果 simpleperf 有 JS 绑定就不需要 C++，如果数据处理脚本是 Java 写的就不需要 Python。

### 2.4 为什么 PM2 Cluster 而不是 K8s 多 Pod

PM2 Cluster（4 个 Node.js 进程在一个容器内）vs K8s 多 Pod（4 个容器各跑一个进程）：

| | PM2 Cluster | K8s 多 Pod |
|---|---|---|
| 进程间通信 | 同机器，Redis localhost | 跨网络，需要云 Redis |
| 文件共享 | 同一个文件系统 | 需要共享存储（PVC） |
| 部署复杂度 | 一个 Dockerfile | 需要 Deployment + Service + PVC |
| 扩展性 | 单机上限 | 无限水平扩展 |

选 PM2 Cluster 的原因：内部工具用户量小（20-30 人），单机 4 进程足够。simpleperf 解析结果缓存在本地磁盘，多 Pod 需要共享存储会增加复杂度。

如果用户量增长，迁移路径：PM2 Cluster → K8s 多 Pod + 云 Redis + 共享 PVC，只需改环境变量和部署配置，代码不用改。

## 三、分层架构的边界划分

```
┌─────────────────────────────────────────────┐
│ 前端应用层                                    │
│ 职责: UI 渲染、用户交互、状态管理              │
│ 边界: 不做数据处理，不直接访问数据库            │
│ 通信: $fetch → BFF 或 Java 后端               │
├─────────────────────────────────────────────┤
│ BFF 层                                       │
│ 职责: 文件操作、Native 计算、进程协调          │
│ 边界: 不做业务逻辑，不直接查数据库             │
│ 通信: 文件系统 + Redis + HTTP 代理             │
├─────────────────────────────────────────────┤
│ Java 后端                                    │
│ 职责: 业务 API、数据查询、文件服务、认证       │
│ 边界: 不做文件解析，不做前端渲染               │
│ 通信: JDBC → Doris, FileInputStream → trace-fs│
├─────────────────────────────────────────────┤
│ Python Consumer                              │
│ 职责: 异步数据解析、格式转换、数据库写入       │
│ 边界: 不提供 API，不做实时查询                 │
│ 通信: Talos MQ 消费 → Doris Stream Load       │
├─────────────────────────────────────────────┤
│ 存储层                                       │
│ Doris: 结构化分析数据                         │
│ FDS/JuiceFS: 原始文件                         │
│ Redis: 缓存 + 协调                            │
└─────────────────────────────────────────────┘
```

每层只做自己该做的事，通过明确的接口通信。这样任何一层可以独立替换——比如把 FDS 换成 S3，只需要改 Java 后端的 FDSService，前端和 Consumer 不用动。

## 四、数据流设计

### 4.1 同步 vs 异步的边界

```
同步路径（用户等待）:
  前端上传 → BFF 接收 → 返回成功
  前端查询 → Java 后端 → Doris 查询 → 返回数据
  前端解析 → BFF C++ Addon → 返回火焰图 JSON

异步路径（后台处理）:
  上传回调 → Talos MQ → Python Consumer → Doris 写入（10-30 分钟）
  前端轮询解析状态 → Java 后端查询 DfpFileRecord.status
```

原则：**用户能感知等待的操作走同步，用户不需要盯着的操作走异步**。

### 4.2 文件流转路径

一个 ZIP 包从上传到最终可视化，经过的路径：

```
用户浏览器
  │ PUT (预签名 URL)
  ▼
FDS 对象存储 (原始 ZIP)
  │ Talos MQ 通知
  ▼
Python Consumer
  │ 从 FDS 下载 ZIP
  │ 解压 → showmap / perf.data / trace
  │ 解析 → CSV / Parquet / HTML
  ▼
┌─────────┬──────────────┐
│ Doris   │ FDS/trace-fs │
│ 结构化  │ 文件类       │
│ 数据    │ (火焰图HTML  │
│         │  trace文件)  │
└────┬────┴──────┬───────┘
     │           │
     ▼           ▼
  Java API    文件服务
     │           │
     ▼           ▼
  前端可视化 (ECharts / DiffTable / FlameGraph / Perfetto)
```

## 五、扩展性设计

当前架构预留了几个扩展点：

| 扩展方向 | 当前状态 | 扩展方式 |
|---------|---------|---------|
| 新增分析维度 | 4 个模块 | 加 pages/ 目录 + Doris 表 + API |
| 多用户并发 | PM2 × 4 | 改 K8s 多 Pod + 云 Redis |
| 数据量增长 | 单 Doris 集群 | Doris 水平扩展（加 BE 节点） |
| 存储统一 | FDS + JuiceFS 并存 | 迁移到统一 S3/JuiceFS |
| 国际化 | 中文 | i18n 插件 |

关键设计：**配置驱动**。菜单路由通过 `menuPath` 配置映射，新增模块只需加配置不需要改框架代码。DiffTable 通过 `IDiffOptions` 配置驱动，新增对比表只需定义配置不需要写新组件。

## 六、架构演进路径

```
V1 (起步): 纯前端
  Nuxt SPA + BFF → 本地文件 → ECharts
  问题: 数据不持久化，换台电脑就没了

V2 (数据持久化): 加入后端
  + Java 后端 + Doris + Python Consumer
  问题: 上传走 BFF 流式接收，大文件慢

V3 (上传优化): FDS 直传
  + FDS 预签名 URL，浏览器直传
  问题: 超过 5GB 传不了

V4 (大文件): MPU 分片上传
  + FDS Multipart Upload
  当前状态

V5 (未来): 存储统一
  FDS → S3/JuiceFS 统一
  并发分片上传，速度提升 6 倍
```

每次演进都是被实际问题驱动的，不是提前设计好的。这是务实的架构演进方式——**解决当前最痛的问题，同时不堵死未来的路**。
