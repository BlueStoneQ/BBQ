# 系统级应用内存合规优化

## 背景

某系统级框架应用（预装 ROM）在 OS4 合规审查中，老化内存峰值（PSS Max）超标。合规要求 PSS Max < 37.3MB，实测 41.9MB，超标 12.3%。该应用为系统常驻进程，承载快应用运行时引擎，dex 体积大（42MB+），代码段 mmap 常驻内存是主要瓶颈。

## 目标

- 老化内存 PSS Max 降至 37.3MB 以下（合规阈值）
- 不影响功能完整性和运行时稳定性
- 方案可复用到后续版本迭代

## 方案

### 核心思路：减少 dex 常驻内存

通过 R8 全局混淆 + DEX Layout Optimization + shrinkResources 三管齐下，缩小 dex 体积，减少运行时代码段 mmap 占用。

### 技术方案

| 优化项 | 原理 | 预期收益 |
|--------|------|---------|
| R8 严格混淆 | 集中在 app 模块做全局混淆，删除未使用代码，缩短类名/方法名 | dex -39% |
| DEX Layout Optimization | 基于 baseline profile 重排 dex 布局，热代码集中在前几页，冷代码延迟加载 | 启动内存降低 |
| shrinkResources | 移除未引用的资源文件 | 资源段缩小 |
| consumerProguardFiles 清理 | 移除 library 模块中阻止 R8 裁剪的宽泛 keep 规则 | R8 生效的前提 |

### 关键技术决策

1. **minifyEnabled 只在 app 模块开启** — library 模块关闭混淆，由 app 统一做全局 R8 分析，效果优于各模块独立混淆
2. **清理 consumerProguardFiles** — 新增模块的宽泛 keep 规则通过 consumer 机制传递给 app，阻止了 R8 裁剪，需逐个清理
3. **保留 consumer 机制** — 只清理宽泛规则，不删除 consumerProguardFiles 声明，保留后续加精细规则的能力

## 行动

### 1. 问题排查

- cherry-pick 已有优化提交后，dex 体积未缩小（44MB → 44MB）
- 分析 R8 seeds.txt 发现 56 万+ keep 条目，几乎所有类被保留
- 定位根因：3 个新增模块的 proguard-rules.pro 中宽泛 keep 规则通过 consumerProguardFiles 传递给 app

### 2. 方案实施

- 涉及 3 个代码仓库、9 笔提交
- 本地验证 dex 从 44.4MB 降至 27.05MB（-39%）
- CI 构建验证通过，产物 4 个 dex / 27.05MB

### 3. ROM 集成验证

- 构建带混淆的 APK → 上传产物仓库 → 生成 manifest → ROM 构建 → 刷机
- 验证 ROM 中 APK 的 dex 结构正确（4 个 dex / 27.05MB）

### 4. 内存实测

- 搭建老化内存测试环境（48 个应用预置 + 数据填充 + 自动化测试工具）
- 执行进程老化模型测试（1000+ 用例，5-6 小时/轮）
- dumpsys meminfo 深度分析内存构成

## 结果

### dex 体积

| | 优化前 | 优化后 | 收益 |
|---|---|---|---|
| dex 文件数 | 6 个 | 4 个 | -2 |
| dex 总大小 | 42.33 MB | 27.05 MB | -15.28 MB（-36%） |

### 内存占用（dumpsys meminfo 对比）

| 类别 | 优化前 | 优化后 | 收益 |
|---|---|---|---|
| .dex mmap | 25.6 MB | 17.8 MB | -7.8 MB |
| System（共享库分摊） | 9.1 MB | 3.1 MB | -6.0 MB |
| **TOTAL PSS** | **43.1 MB** | **33.4 MB** | **-9.7 MB（-22.5%）** |

### 合规达标

| 指标 | 优化前 | 优化后 | 阈值 |
|---|---|---|---|
| 空闲 PSS | 43.1 MB ❌ | 33.4 MB ✅ | < 37.3 MB |
| 老化 PSS Max | 待最终数据 | 待最终数据 | < 37.3 MB |

> 老化测试进行中，空闲状态已达标，预期老化 Max 在 31-35MB 区间。
