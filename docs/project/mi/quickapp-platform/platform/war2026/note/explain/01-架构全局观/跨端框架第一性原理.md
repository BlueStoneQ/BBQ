## 一、根本问题：为什么需要跨端？

### 原生开发的痛点

原生开发（Android/iOS 各写一套）的根本问题不是"写两遍代码"，而是：

**开发效率 × 发布效率 × 维护成本 的乘积太高。**

| 痛点 | 本质 |
|------|------|
| 写两遍 | 人力成本 ×2 |
| 发版慢 | 应用商店审核 1-7 天，热修复受限 |
| 安装门槛 | 用户不愿为轻量需求下载 100MB 的 app |
| 一致性差 | 两端 UI/逻辑容易不一致 |

### 第一性原理推导

如果我们从零设计，要解决"一份代码运行在多个平台"，需要回答一个根本问题：

**代码在哪一层统一？**

```
应用代码（业务逻辑 + UI 描述）
        │
        ▼
   ┌─────────────┐
   │  统一层在哪？ │  ← 这是所有跨端方案的核心分歧
   └─────────────┘
        │
        ▼
平台能力（渲染、系统 API、硬件）
```

不同方案选择在不同层统一：

| 统一层 | 方案 | 代表 |
|--------|------|------|
| 在 UI 描述层统一 | 用 Web 标准（HTML/CSS/JS）描述 UI，各平台各自渲染 | 快应用、RN |
| 在渲染层统一 | 自带渲染引擎，绕过平台 UI 系统 | Flutter |
| 在运行时统一 | 用 WebView 直接跑 Web 代码 | Cordova、小程序 |
| 在编译层统一 | 编译时转换为各平台原生代码 | Kotlin Multiplatform |

## 二、本质 Tradeoff：不可能三角

跨端框架存在一个**不可能三角**：

```
        性能
       /    \
      /      \
     /        \
  开发效率 ── 平台一致性
```

- **性能**：渲染帧率、启动速度、内存占用
- **开发效率**：一份代码、热更新、Web 技术栈
- **平台一致性**：各平台表现完全一致

**你最多只能优化两个，第三个必然妥协：**

| 方案 | 优化了 | 妥协了 |
|------|--------|--------|
| WebView（Cordova） | 开发效率 + 一致性 | 性能（WebView 渲染慢） |
| Native 渲染（RN/快应用） | 性能 + 开发效率 | 一致性（各平台 Native 组件表现不同） |
| 自绘引擎（Flutter） | 性能 + 一致性 | 开发效率（新语言 Dart、不能热更新） |

## 三、四种解法与约束

### 方案 1：WebView 容器

```
JS/HTML/CSS → WebView 渲染 → 屏幕
```

**本质**：把浏览器嵌入 App，Web 页面就是 UI。

**约束**：
- WebView 渲染性能差（DOM → Layout → Paint → Composite 全在一个进程）
- 无法直接访问系统能力（需要 JSBridge 中转）
- 内存占用高（一个 WebView ≈ 一个浏览器 Tab）

**适用场景**：对性能要求不高的信息展示类应用。

### 方案 2：Native 渲染 + JS 逻辑

```
JS 逻辑 → Bridge → Native View 渲染 → 屏幕
```

**本质**：用 JS 写逻辑和 UI 描述，但渲染交给平台原生 View 系统。

**约束**：
- Bridge 通信有开销（序列化/反序列化、线程切换）
- 各平台 Native 组件行为不完全一致
- 需要维护一套 JS → Native 的映射层

**适用场景**：需要接近原生性能 + Web 开发效率的应用。**快应用、React Native 都是这个方案。**

### 方案 3：自绘引擎

```
Dart 逻辑 → Skia 渲染 → 屏幕（绕过平台 View 系统）
```

**本质**：自带渲染引擎（Skia），直接操作 GPU 画像素，不依赖平台 UI 组件。

**约束**：
- 无法复用平台 UI 组件（需要自己实现所有控件）
- 包体大（引擎本身 4-5 MB）
- 不支持官方热更新
- 新语言学习成本

**适用场景**：对 UI 一致性要求极高、性能敏感的应用。

### 方案 4：编译时转换

```
Kotlin/共享代码 → 编译 → 各平台原生代码
```

**本质**：在编译阶段把共享逻辑转换为各平台的原生实现。

**约束**：
- UI 层通常不能共享（各平台 UI 框架差异太大）
- 调试困难（编译产物和源码不一致）

**适用场景**：共享业务逻辑层，UI 各写各的。

## 四、场景决定方案

### 场景：手机厂商想让轻量应用免安装运行

**约束条件**：
1. 必须免安装（不能让用户下载 APK）
2. 必须接近原生性能（用户体验不能比原生 App 差太多）
3. 必须能访问系统能力（相机、GPS、通知等）
4. 必须轻量（包体小、启动快）
5. 开发者生态要大（最好用 Web 技术栈）

**排除法**：
- ❌ WebView：性能不够（约束 2 不满足）
- ❌ Flutter：需要安装（约束 1 不满足）、包体大（约束 4 不满足）、开发者少（约束 5 不满足）
- ❌ 编译时转换：不支持热更新/免安装（约束 1 不满足）
- ✅ **Native 渲染 + JS 逻辑**：免安装（RPK 包动态下载）、原生性能、Web 技术栈

**这就是快应用选择 Native 渲染方案的根本原因。**

### 场景：JS 要调用系统能力（相机/GPS）

**为什么不直接暴露 Native API 给 JS？**

因为 JS 运行在 V8 引擎里，V8 是一个沙箱——它只认识 JS 对象，不认识 Android 的 `Camera` 类。

**Bridge 的本质**：在两个不同的类型系统之间建立映射。

```
JS 世界                    Native 世界
─────────                  ─────────
Object/String/Number  ←→   Java Object/String/int
function callback     ←→   Callback ID（数字）
异步 Promise          ←→   线程池 + Handler 投递
```

**为什么用回调 ID 而不是直接传函数？**

因为 V8 的函数对象无法序列化传递给 Java。函数是一个闭包，包含执行上下文、作用域链，这些东西在 Java 侧没有对应物。所以只能用 ID 做映射——JS 侧存函数，Native 侧存 ID，回调时用 ID 找回函数。

### 场景：要同时运行多个轻应用且互不影响

**为什么不用多线程？**

1. **V8 是单线程的**：一个 V8 实例只能在一个线程上执行 JS。如果两个快应用共享一个 V8 实例，它们的 JS 代码会互相阻塞。
2. **内存隔离**：线程共享进程内存，一个快应用的内存泄漏会影响所有快应用。
3. **崩溃隔离**：一个线程崩溃（Native Crash）会导致整个进程崩溃，所有快应用一起死。

**多进程的代价**：
- 进程创建开销（fork + Application 初始化）
- IPC 通信开销（Binder）
- 内存占用增加（每个进程有独立的虚拟地址空间）

**快应用的折中**：固定 5 个 Launcher 进程（0~4），通过数据库分配槽位复用，避免频繁创建/销毁进程。

## 五、快应用的设计选择

| 设计决策 | 选择 | 原因 |
|----------|------|------|
| 渲染方式 | Native View | 性能接近原生，免安装 |
| JS 引擎 | V8 | 性能最好的 JS 引擎 |
| 通信方式 | J2V8 直接绑定（JSI） | 同步调用、零序列化开销 |
| 进程模型 | 多进程（5 个 Launcher） | 隔离性 + 并发 |
| 布局引擎 | Yoga（Flexbox） | Web 开发者熟悉 |
| 包格式 | RPK（zip + 签名） | 轻量、可流式加载 |
| 分发方式 | 系统预装 + 应用市场 | 免安装、系统级入口 |
| 开发语言 | HTML/CSS/JS（.ux 文件） | Web 生态、开发者多 |

## 六、核心设计问题拆解

学习一个跨端框架，本质上是在回答这些设计问题：

| # | 问题 | 快应用的答案 | 对应 Phase |
|---|------|-------------|-----------|
| 1 | 应用怎么启动？ | Intent → Dispatcher → Launcher → V8 | Phase 2 |
| 2 | JS 怎么调用系统能力？ | V8 绑定 → ExtensionManager → Feature | Phase 3, 5 |
| 3 | UI 怎么渲染？ | 虚拟 DOM → Action → VDomActionApplier → Native View | Phase 4 |
| 4 | 应用怎么分发？ | RPK 包 → DistributionManager → Cache | Phase 6 |
| 5 | 性能怎么保证？ | 多进程、批量 Action、V8 Snapshot、R8 | Phase 7 |
| 6 | 多个应用怎么隔离？ | Launcher0~4 独立进程 | Phase 2 |
| 7 | 怎么和系统集成？ | HostService（AIDL）、Widget、ContentProvider | Phase 2 |

每个问题都可以继续追问"为什么这样设计？不这样会怎样？"——这就是后续 Phase 的内容。
