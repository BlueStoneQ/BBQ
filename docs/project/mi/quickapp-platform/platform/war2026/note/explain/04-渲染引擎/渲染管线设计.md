## 一、核心问题

| 问题 | 本质 |
|------|------|
| JS 的 DOM 操作怎么映射到 Native View？ | 指令序列化 + 异步应用 |
| 为什么不直接操作 View？ | JS 线程和 UI 线程不同，需要跨线程通信 |
| 批量发送 Action 解决什么问题？ | 减少 Bridge 调用次数，类似浏览器的 requestAnimationFrame |
| 组件注册表怎么工作？ | 标签名 → Widget 类的映射（编译时生成） |

## 二、渲染的本质

**第一性原理**：渲染 = 把数据结构（树）转换为像素。

```
数据（状态）→ UI 描述（树）→ 平台渲染指令 → 像素
```

不同方案在不同环节切入：
- **浏览器**：HTML DOM → Layout → Paint → Composite
- **Flutter**：Widget Tree → RenderObject → Skia → GPU
- **快应用**：虚拟 DOM → Action 指令 → Android View Tree → 硬件加速渲染

快应用的特殊之处：**UI 描述在 JS 线程，渲染在 UI 线程**，中间需要一个"指令队列"来桥接。

## 三、三级管线

### 第一级：编译时（.ux → JS Bundle）

```
<text>Hello {{name}}</text>
    ↓ hap-compiler
$app_template$ = { type:'text', attr:{ value: () => 'Hello ' + this.name } }
```

**为什么在编译时做？** 运行时不需要解析 HTML，直接构建虚拟 DOM，省去解析开销。

### 第二级：运行时 JS 层（虚拟 DOM → Action）

```
数据变化 → DSL 响应式更新 → Listener 捕获 DOM 操作 → Streamer 批量缓冲 → callNative()
```

关键组件：
- **Listener**：监听所有 DOM 操作（add/remove/update），转换为标准 Action 格式
- **Streamer**：缓冲 Action，达到 50 条或页面渲染完成时批量发送

### 第三级：运行时 Native 层（Action → View）

```
callNative() → RenderActionManager（IO 线程解析 JSON）→ VDomActionApplier（UI 线程应用变更）
```

关键组件：
- **RenderWorker**：IO 线程，解析 JSON、计算样式、构建 VDomChangeAction
- **VDomActionApplier**：UI 线程，根据 Action 创建/更新/删除 Android View
- **ComponentFactory**：标签名 → Widget 类的映射表

## 四、关键设计决策

### 为什么 JSON 解析放在 IO 线程？

**场景**：一个复杂页面首次渲染，可能有 500+ 个 Action（创建 500 个 View）。

如果在 UI 线程解析 JSON：
- 解析 500 个 Action 的 JSON 可能需要 50-100ms
- UI 线程被阻塞，用户看到卡顿

放在 IO 线程：
- JSON 解析和样式计算在后台完成
- UI 线程只做 View 创建和挂载（必须在 UI 线程的操作）
- 两者可以流水线并行

### 为什么用 VDocument 而不是直接操作 View？

**VDocument 是 Native 侧的虚拟 DOM**，维护了 vId → VElement 的映射。

作用：
1. **快速查找**：JS 侧通过 ref（vId）引用节点，Native 侧通过 HashMap O(1) 查找
2. **批量更新**：先更新 VDocument，再一次性同步到 View Tree
3. **Inspector 支持**：调试器需要遍历 DOM 树，VDocument 提供了这个能力

### 为什么组件用注解注册？

```java
@WidgetAnnotation(name = "text", ...)
public class Text extends AbstractText<TextLayoutView> { ... }
```

编译时通过 APT（Annotation Processing Tool）扫描所有 `@WidgetAnnotation`，生成注册表代码。运行时通过标签名查表即可，不需要反射。

**好处**：
- 编译时检查（标签名重复会报错）
- 运行时零反射开销
- 支持第三方扩展（只要加注解就能注册新组件）

## 五、与浏览器渲染的对比

| 环节 | 浏览器 | 快应用 |
|------|--------|--------|
| UI 描述 | HTML DOM | 虚拟 DOM（JS 对象） |
| 变更检测 | MutationObserver | Listener |
| 指令队列 | 渲染队列（微任务后 flush） | Streamer（50 条阈值） |
| 布局计算 | CSS Layout（Blink） | Yoga Flexbox |
| 渲染执行 | Paint + Composite | Android View measure/layout/draw |
| 硬件加速 | GPU Composite Layer | RenderThread + GPU |

**核心差异**：浏览器的 DOM 和渲染在同一进程同一线程（主线程），快应用的 DOM 在 JS 线程、渲染在 UI 线程，中间多了一层跨线程通信。

## 六、Android 知识映射

| 框架概念 | Android 知识点 |
|----------|---------------|
| VDomActionApplier | ViewGroup.addView() / removeView() |
| ComponentFactory | 工厂模式 + 注解处理器（APT） |
| Yoga Flexbox | LayoutParams、measure/layout 流程 |
| RenderWorker | Executors.io() 线程池 |
| 批量更新 | 类比 RecyclerView 的 DiffUtil |
| VDocument | 类比 Accessibility Node Tree |
