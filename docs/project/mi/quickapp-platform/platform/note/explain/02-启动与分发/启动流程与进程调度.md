## 二、启动全链路

```
外部触发（deeplink/应用市场/Widget/Push）
    │
    ▼
VendorDispatcherActivity（透明 Activity，不渲染 UI）
    │  ① onPreDispatch()：预热目标 Launcher 进程
    │  ② setSource()：解析来源信息（谁调起的、从哪个渠道）
    │  ③ dispatch()：路由决策
    │
    ▼
VendorDispatcherHelper.launch()
    │  ④ EntranceController.checkForLaunching()：云控策略检查
    │     - 频率限制（同一个快应用 N 秒内只能拉起 M 次）
    │     - 黑名单（广告快应用禁止拉起）
    │     - 白名单（系统应用不受限制）
    │
    ▼
LauncherManager.launch()
    │  ⑤ Launcher.select(pkg)：为目标快应用分配 Launcher 槽位
    │  ⑥ intent.setClassName("VendorLauncherActivity$Launcher" + id)
    │
    ▼
VendorLauncherActivity$Launcher1（独立进程 :Launcher1）
    │  ⑦ 显示 Splash 页
    │  ⑧ DistributionManager 检查 RPK 状态（已安装/需下载/需更新）
    │  ⑨ 加载 RPK → 创建 V8 引擎 → 执行 JS → 渲染第一帧
    │
    ▼
用户看到快应用页面
```

## 三、关键设计决策

### 为什么 Dispatcher 和 Launcher 分开？

**场景**：用户点击一个快应用链接，但这个快应用还没下载。

如果合并成一个 Activity：
- 下载期间这个 Activity 占着前台，用户看到白屏
- 下载完成后要在同一个 Activity 里切换状态（loading → 渲染），逻辑复杂

分开后：
- Dispatcher 只做路由决策（毫秒级），立即 finish
- Launcher 负责完整的应用生命周期（下载 → 安装 → 运行）
- Launcher 可以显示 Splash、Loading、错误页等不同状态

**本质**：单一职责原则。Dispatcher = 路由器，Launcher = 应用容器。

### 为什么 Dispatcher 是透明 Activity 而不是 Service？

因为启动快应用需要 **Activity 上下文**：
- 需要 `startActivity()` 启动 Launcher（从 Service 启动 Activity 需要 `FLAG_ACTIVITY_NEW_TASK`，行为不可控）
- 需要 `getCallingPackage()` 获取调用方信息
- 需要处理 Intent Filter 匹配（deeplink 只能匹配到 Activity）

### 为什么预热进程（onPreDispatch）？

**场景**：用户点击快应用链接，如果 Launcher 进程不存在，需要 fork 新进程 → Application.onCreate() → Activity.onCreate()，这个过程 200-500ms。

预热策略：在 Dispatcher 做路由决策的同时，异步启动目标 Launcher 进程的 Service，让进程提前创建好。等 Dispatcher 决策完成后，Launcher Activity 启动时进程已经存在，省去 fork 时间。

## 四、多进程 Launcher 调度

### 槽位分配算法

```java
// Launcher.select(context, pkg)
// 通过 ContentProvider 查询 LauncherTable 数据库
// 返回 LauncherInfo { id, pkg, isAlive, activeAt }
```

分配逻辑：
1. 如果该 pkg 已有分配的槽位 → 复用
2. 如果有空闲槽位 → 分配新的
3. 如果所有槽位都被占用 → 选择最久未活跃的（LRU）

### 为什么是 5 个？

- 太少（1-2 个）：无法并发运行多个快应用
- 太多（10+）：每个进程占 30-50 MB 内存，10 个就是 300-500 MB
- 5 个是经验值：覆盖绝大多数用户场景（同时打开 5 个快应用已经很极端）

### 进程命名

```
com.miui.hybrid:Launcher0
com.miui.hybrid:Launcher1
com.miui.hybrid:Launcher2
com.miui.hybrid:Launcher3
com.miui.hybrid:Launcher4
```

每个进程有独立的 V8 引擎实例、独立的内存空间、独立的 Activity 栈。

## 五、云控与入口管理

### EntranceController 解决什么问题？

**场景**：某个广告快应用通过 Push 每分钟拉起一次，用户体验极差。

**方案**：在 Dispatcher 层加一道"门禁"——EntranceController，根据云端下发的规则决定是否允许拉起。

规则示例：
```json
{
  "controlStatus": 1,
  "pkg": ["com.ad.spam.*"],
  "channels": ["push"],
  "maxFrequency": "3/hour"
}
```

### 入口来源（Source）追踪

每次启动都会记录来源信息：
- 谁调起的（packageName）
- 从哪个渠道（应用市场/负一屏/Push/deeplink/Widget）
- 场景（shortcut/intent/host_sdk）

用于：统计分析、云控决策、广告归因。
