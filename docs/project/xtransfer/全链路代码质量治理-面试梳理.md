## 一、项目背景与价值

### 1.1 业务痛点
- **代码质量参差不齐**：团队成员编码风格不统一，代码可读性差
- **低级错误频发**：语法错误、类型错误等问题频繁出现
- **Code Review 效率低**：大量时间浪费在格式和规范问题上
- **线上故障率高**：代码质量问题导致线上 bug 频发
- **技术债务累积**：缺乏统一规范，代码维护成本越来越高

### 1.2 解决方案
建立**全链路代码质量治理体系**，从编码、提交、CI/CD 到 Code Review 全流程卡控代码质量。

### 1.3 核心价值
- **提升代码质量**：统一编码规范，减少低级错误
- **提高开发效率**：自动化检查和修复，减少人工 Review 成本
- **降低故障率**：多层卡控，确保代码质量
- **技术债务可控**：持续监控和改进代码质量

## 二、系统架构设计

```

├─────────────────────────────────────────────────────┤
│                                                       │

│  │         第一层：编码时（实时反馈）             │  │
│  │  ESLint 插件 | Prettier | 实时提示 | 自动修复 │  │
│  └──────────────────────────────────────────────┘  │

│  │         第二层：提交时（Git Hooks）            │  │

│  └──────────────────────────────────────────────┘  │

│  │         第三层：CI 阶段（GitLab CI）           │  │

│  └──────────────────────────────────────────────┘  │
│                      ↓                               │
│  ┌──────────────────────────────────────────────┐  │
│  │         第四层：Code Review                    │  │
│  │  领域 A/B 角 | 人工审查 | 架构评审            │  │

└─────────────────────────────────────────────────────┘
```

### 2.2 核心组成

#### 2.2.1 规范体系
- **JavaScript**：ESLint + Standard 规范
- **TypeScript**：TSLint / ESLint + TS 插件
- **CSS**：StyleLint
- **Vue**：eslint-plugin-vue
- **React**：eslint-plugin-react

#### 2.2.2 卡控体系
- **编码时**：IDE 插件实时提示和自动修复
- **提交时**：Git Hooks 增量检查
- **CI 阶段**：全量检查 + 单元测试
- **Code Review**：人工审查

### 3.2 提交时质量卡控（Git Hooks）

#### 3.2.1 Husky + Lint-staged 配置

**安装依赖**：

```bash
npm install husky lint-staged --save-dev
npx husky install
```

**package.json 配置**：

```json
{
  "scripts": {
    "prepare": "husky install"
  },
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": [
      "eslint --fix",
      "prettier --write",
      "git add"
    ],
    "*.{css,scss,less}": [
      "stylelint --fix",
      "prettier --write",
      "git add"
    ],
    "*.vue": [
      "eslint --fix",
      "prettier --write",
      "git add"
    ]
  }
}
```

**创建 pre-commit Hook**：

```bash
npx husky add .husky/pre-commit "npx lint-staged"
```

**pre-commit 文件内容**：

```bash
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"

echo "🔍 Running lint-staged..."
npx lint-staged

if [ $? -ne 0 ]; then
  echo "❌ Lint failed. Please fix the errors and try again."
  exit 1
fi

echo "✅ Lint passed!"
```

**commitlint 配置**：


**commitlint.config.js**：

```javascript
module.exports = {

      [

```


### 3.3 CI 阶段质量卡控

#### 3.3.1 GitLab CI 配置

**.gitlab-ci.yml**：

```yaml
stages:
  - lint
  - test
  - build

# 代码检查
lint:
  stage: lint
  image: node:16
  script:
    - npm ci
    - npm run lint
  only:
    - merge_requests
    - master
  allow_failure: false

# 单元测试
test:
  stage: test
  image: node:16
  script:
    - npm ci
    - npm run test:coverage
  coverage: '/All files[^|]*\|[^|]*\s+([\d\.]+)/'
  artifacts:
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml
  only:
    - merge_requests
    - master
  allow_failure: false

# 构建
build:
  stage: build
  image: node:16
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - dist/
  only:
    - master
```

#### 3.3.2 Jenkins 配置

**Jenkinsfile**：

```groovy
pipeline {
  agent any
  
  stages {
    stage('Install') {
      steps {
        sh 'npm ci'
      }
    }
    
    stage('Lint') {
      steps {
        sh 'npm run lint'
      }
    }
    
    stage('Test') {
      steps {
        sh 'npm run test:coverage'
      }
      post {
        always {
          junit 'coverage/junit.xml'
          publishHTML([
            reportDir: 'coverage',
            reportFiles: 'index.html',
            reportName: 'Coverage Report'
          ])
        }
      }
    }
    
    stage('Build') {
      steps {
        sh 'npm run build'
      }
    }
  }
  
  post {
    failure {
      // 发送通知
      emailext(
        subject: "Build Failed: ${env.JOB_NAME} - ${env.BUILD_NUMBER}",
        body: "Check console output at ${env.BUILD_URL}",
        to: "${env.CHANGE_AUTHOR_EMAIL}"
      )
    }
  }
}
```

### 3.4 Code Review 流程

#### 3.4.1 Review 规范

**必须 Review 的内容**：
- 代码逻辑正确性
- 边界条件处理
- 错误处理
- 性能问题
- 安全问题
- 可维护性

**Review 清单**：

```markdown
## Code Review Checklist

### 功能性
- [ ] 代码实现符合需求
- [ ] 边界条件处理完善
- [ ] 错误处理合理

### 代码质量
- [ ] 命名清晰易懂
- [ ] 逻辑简洁明了
- [ ] 无重复代码
- [ ] 注释充分

### 性能
- [ ] 无明显性能问题
- [ ] 算法复杂度合理
- [ ] 资源使用合理

### 安全
- [ ] 无安全漏洞
- [ ] 输入验证完善
- [ ] 敏感信息处理得当

### 测试
- [ ] 单元测试覆盖充分
- [ ] 测试用例合理
```

#### 3.4.2 A/B 角机制

**角色定义**：
- **A 角**：领域负责人，深度 Review
- **B 角**：领域备份，交叉 Review

**Review 流程**：
1. 开发者提交 MR（Merge Request）
2. A 角进行首次 Review
3. B 角进行交叉 Review
4. 两人都 Approve 后才能合并

## 四、技术难点与解决方案

### 4.1 规范冲突

**难点**：ESLint 和 Prettier 规则冲突

**解决方案**：
- 使用 `eslint-config-prettier` 关闭冲突规则
- Prettier 负责格式化，ESLint 负责代码质量
- 统一配置，避免团队成员配置不一致

### 4.2 性能问题

**难点**：全量检查耗时长，影响开发体验

**解决方案**：
- **编码时**：只检查当前文件
- **提交时**：只检查暂存区文件（lint-staged）
- **CI 阶段**：全量检查，但并行执行

### 4.3 历史代码

**难点**：存量代码不符合规范，一次性修复成本高

**解决方案**：
- **增量治理**：只检查新增和修改的代码
- **逐步迁移**：按模块逐步修复历史代码
- **白名单机制**：暂时忽略部分历史文件

### 4.4 团队推广

**难点**：团队成员抵触，执行不到位

**解决方案**：
- **培训宣讲**：讲解规范的必要性和好处
- **工具支持**：提供自动化工具，降低使用成本
- **强制卡控**：通过 CI 强制执行，无法绕过
- **持续优化**：根据反馈调整规则

## 五、面试高频问题

### 5.1 为什么要做代码质量治理？

**回答要点**：
- **提升代码质量**：统一规范，减少低级错误
- **提高开发效率**：自动化检查，减少 Review 成本
- **降低故障率**：多层卡控，确保代码质量
- **技术债务可控**：持续监控和改进

### 5.2 全链路治理包括哪些环节？

**回答要点**：
- **编码时**：IDE 插件实时提示和自动修复
- **提交时**：Git Hooks 增量检查
- **CI 阶段**：全量检查 + 单元测试
- **Code Review**：人工审查

### 5.3 如何保证 Git Hooks 不被绕过？

**回答要点**：
- **CI 强制检查**：即使绕过 Hooks，CI 也会检查
- **团队规范**：明确要求不能使用 `--no-verify`
- **Code Review**：人工审查时发现问题
- **监控告警**：监控绕过行为

### 5.4 ESLint 和 Prettier 有什么区别？

**回答要点**：
- **ESLint**：代码质量检查（逻辑错误、最佳实践）
- **Prettier**：代码格式化（缩进、换行、引号）
- **配合使用**：Prettier 格式化，ESLint 检查质量
- **避免冲突**：使用 `eslint-config-prettier`

### 5.5 如何处理历史代码？

**回答要点**：
- **增量治理**：只检查新增和修改的代码
- **逐步迁移**：按模块逐步修复
- **白名单机制**：暂时忽略部分文件
- **设定目标**：制定迁移计划和时间表

### 5.6 如何推动团队执行？

**回答要点**：
- **培训宣讲**：讲解必要性和好处
- **工具支持**：降低使用成本
- **强制卡控**：CI 强制执行
- **持续优化**：根据反馈调整

### 5.7 单元测试覆盖率要求多少？

**回答要点**：
- **核心模块**：80% 以上
- **一般模块**：60% 以上
- **工具函数**：90% 以上
- **逐步提升**：设定阶段性目标
