## RAG 的本质

**RAG（Retrieval-Augmented Generation）= 检索增强生成**

一句话：LLM 不知道的事，先帮它查到，塞进 prompt 里，让它"有据可依"地回答。

```
没有 RAG：
  用户问 → LLM 凭记忆回答（可能幻觉、可能过时）

有 RAG：
  用户问 → 先从知识库检索相关文档 → 塞进 prompt → LLM 基于文档回答（有据可依）
```

**本质就是"动态扩充上下文"**——LLM 的知识是训练时固定的，RAG 让它能访问外部最新信息。

## RAG 的完整流程

```
┌─────────────────────────────────────────────────────────┐
│                    离线阶段（建索引）                      │
│                                                         │
│  文档 → 切分 Chunk（500-1000 字一段）                    │
│      → 每个 Chunk 调 Embedding 模型 → 得到向量（[0.1, 0.3, ...] 1536维）│
│      → 存入向量数据库（Pinecone / Milvus / Chroma）       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    在线阶段（查询）                        │
│                                                         │
│  用户问题 → 调 Embedding 模型 → 得到问题向量              │
│          → 在向量数据库中找最相似的 Top-K 个 Chunk         │
│          → 把这些 Chunk 拼接到 prompt 里                  │
│          → 发给 LLM 生成回答                             │
└─────────────────────────────────────────────────────────┘
```

### 关键概念

| 概念 | 一句话 |
|------|--------|
| **Embedding** | 把文本变成向量（一组数字，如 1536 维浮点数）。语义相似的文本，向量距离近。注意：不是"嵌入到服务器"，是"映射到向量空间" |
| **Embedding 模型** | 专门做"文本→向量"转换的模型（如 OpenAI text-embedding-3-small）。它不生成文字，只做转换。可以调 API（主流，很便宜，比 LLM 便宜 100 倍），也可以跑本地模型（sentence-transformers / Ollama，几百 MB，普通电脑能跑）。比 LLM 小很多，因为任务简单——只需"理解语义算出一组数字"，不需要生成 |
| **向量数据库** | 本质是建立了"向量→原始文本"的索引。每条记录 = 向量（算相似度用）+ 原始文本（给 LLM 看）+ 元数据（来源等）。核心能力：给一个查询向量，快速找到最相似的 K 个，返回对应原文。和 MySQL 对比：MySQL 按字段精确查（`WHERE name='xxx'`），向量数据库按"语义距离"模糊查（"跟这个向量最像的 5 个"） |
| **Chunk** | 文档切分后的小段（文档太长不能整个 embed，需要切成 500-1000 字的段） |
| **Chunk 切分** | 常用工具：LangChain TextSplitter、LlamaIndex NodeParser。策略：按固定字符数（最简单）、按 token 数（精确）、按段落换行（Markdown）、按语义断点（效果最好成本高）、按代码结构（函数/类）。关键参数：`chunk_size`（每段多大）+ `chunk_overlap`（相邻段重叠 10-20%，防止关键信息被切断） |
| **Top-K** | 取相似度最高的 K 个结果 |
| **余弦相似度** | 衡量两个向量方向是否一致（1=完全相同，0=无关） |

### RAG 核心部件

| 部件 | 作用 | 业界常用 |
|------|------|---------|
| **Embedding 模型** | 文本→向量 | OpenAI `text-embedding-3-small`、Cohere、BGE（国产开源） |
| **向量数据库** | 存储和检索向量 | Pinecone（云服务）、Milvus（开源）、Chroma（轻量本地）、Weaviate、Qdrant |
| **文档处理** | 加载各种格式 + 切分 Chunk | LangChain DocumentLoader、LlamaIndex、Unstructured |
| **Chunk 策略** | 怎么切文档（切太大→检索不精确，切太小→丢上下文） | 按段落 / 按 token 数 / 按语义切分 |

### 完整流程（重要！理解本质）

```
=== 离线阶段（建索引，只做一次）===

原始文档（PDF/Markdown/网页）
  → 文档加载（PDF→纯文本）
  → Chunk 切分（切成 N 段，每段 500 字）
  → 每段调 Embedding 模型 → 得到 N 个向量
  → N 个向量 + 原始文本 一起存入向量数据库
    （向量是索引，原始文本是内容，都要存）

=== 在线阶段（每次查询）===

用户输入 "支付超时怎么处理"
  → 调 Embedding 模型 → 得到问题向量
  → 向量数据库查找最相似的 Top-K 个向量
  → 拿回的是对应的原始文本（不是向量！LLM 不认识向量）
  → 把这些文本拼接到 prompt 里
  → 发给 LLM 生成回答

关键：向量只是"索引"，帮你找到相关文本。最终喂给 LLM 的还是人类可读的文字。
类比：图书馆索引卡（向量）帮你找到书（文本），你借的是书不是索引卡。
```

### 一个具体例子

```
知识库：公司内部 200 篇技术文档

用户问："我们的支付系统怎么处理超时？"

1. Embedding("我们的支付系统怎么处理超时？") → [0.12, 0.45, 0.78, ...]
2. 向量数据库查找最相似的 3 个 Chunk：
   - Chunk A："支付超时重试策略：指数退避..."（相似度 0.92）
   - Chunk B："支付网关超时配置为 30s..."（相似度 0.87）
   - Chunk C："订单超时自动取消逻辑..."（相似度 0.81）
3. 拼接到 prompt：
   "基于以下参考资料回答用户问题：\n{Chunk A}\n{Chunk B}\n{Chunk C}\n\n用户问题：..."
4. LLM 基于这些资料生成准确回答
```

## 向量检索 vs BM25

| 维度 | 向量检索（Embedding） | BM25（关键词） |
|------|---------------------|---------------|
| **原理** | 语义相似度（向量距离） | 关键词匹配频率（TF-IDF 改进） |
| **优势** | 理解语义（"狗"和"犬"能匹配） | 精确匹配（文件路径、函数名） |
| **劣势** | 需要 Embedding 模型 + 向量库 | 不懂语义（"快乐"和"开心"匹配不上） |
| **依赖** | Embedding API + 向量数据库 | 零依赖，纯本地计算 |
| **适合** | 自然语言问答、文档检索 | 代码搜索、精确关键词场景 |
| **成本** | 每次检索要调 API（或本地模型） | 免费，CPU 计算 |

**Mako 选 BM25 的理由**：代码场景下搜索的是文件路径、函数名、变量名——这些是精确关键词，BM25 够用。不需要语义理解。

## 为什么 Mako 不用 RAG

| 原因 | 展开 |
|------|------|
| **时序性** | 编码上下文是有序的——第 3 轮的决策依赖第 2 轮的结果。向量检索按"相似度"排序，不保证时间顺序。RAG 检索回来的 Chunk 可能是 [第4轮, 第2轮, 第3轮]，LLM 不知道"报错是改完之后才出的"，可能做出错误判断。BM25 同样不天然保证时序，但 Mako L5 按时间顺序归档，检索回来后按 timestamp 排序再塞回 messages，成本很低。向量检索做这个更麻烦——得额外存 metadata 再手动排序 |
| **部署复杂** | 需要 Embedding 模型（每次调 API 或跑本地模型）+ 向量数据库。Mako 是 CLI 工具，client 端运行，不能依赖外部服务 |
| **场景不匹配** | RAG 适合"从海量文档中检索知识"；Mako 需要的是"管理对话历史"——历史消息是线性的、有时序的，不是散落的知识片段 |
| **渐进式够用** | L1-L4 解决 80% 的情况（零成本或一次 LLM 调用）。L5 用 BM25 做轻量检索作为兜底，不需要向量检索的重炮 |

**话术**：
> "RAG 解决的是'从海量无序知识中检索'的问题，但 Agent 的上下文是有序的对话历史。我用渐进式管道——大部分情况前 4 层就解决了，L5 用 BM25 关键词检索做兜底，对代码场景够用。如果未来需要接入项目文档知识库，架构上 L5 可以替换为向量检索，不改上层代码。"

## LangChain 是什么

**一句话**：最流行的 LLM 应用开发框架。提供一整套抽象：Chain（链式调用）、Agent（自主决策）、Memory（记忆）、Retriever（检索）、Tool（工具）。

**它解决的问题**：
- 快速原型——10 分钟跑通一个 RAG demo 或 Agent demo
- 多模型切换——统一接口适配 100+ 模型
- RAG 全链路——文档加载 → 切分 → Embedding → 向量库 → 检索 → 拼接 → LLM

**它的核心抽象**：

```
LangChain 架构：
├── LLM / ChatModel        — 模型调用
├── PromptTemplate         — 提示词模板
├── Chain                  — 链式调用（输入→处理→输出）
├── Agent                  — 自主决策循环
├── Memory                 — 对话记忆（BufferMemory / SummaryMemory）
├── Retriever              — 检索器（向量检索 / BM25）
├── Tool                   — 工具定义
├── OutputParser           — 输出解析
└── Callback               — 生命周期钩子
```

## 为什么 Mako 不用 LangChain

> **常见误解**：是不是因为 Mako 是本地 Agent 所以不能用 LangChain？不是。LangChain 也能在本地跑（它是 Python/JS 库，不是云服务）。真正的原因是控制粒度和阶段匹配的问题。

### 核心原因：阶段不匹配

> LangChain 解决的是"从 0 到 1 快速搭原型"的问题。Mako/Claude Code 面对的是"从 1 到 100 精细打磨"的问题。**这两个阶段需要的东西完全不同。**

类比：
- LangChain = 预制板房（快速盖起来，但改结构很难）
- Mako = 自建房（慢一点起步，但每面墙都能按需调整）

AI 时代"自建"的速度已经很快了（和 AI 一起写，几小时搞定核心循环），所以预制板房的"快"优势缩水了，但"改不动"的劣势没变。

### 控制粒度的矛盾

LangChain 做了大量"替你决策"的事——Memory 怎么管、工具怎么调、循环怎么转。这些恰恰是 Coding Agent 需要**自己精细控制**的核心逻辑：

```
LangChain 模式：你配置参数，框架帮你跑（控制权在框架）
Mako/CC 模式：每一个 token 怎么管、每个工具确认流程怎么走、都要自己决定（控制权在你）
```

当你想做框架没预设的事（比如 AsyncGenerator 双向通信、5 层上下文管道、危险工具三级确认），要花大量时间"绕过"框架抽象。这个绕的成本，比自己写还高。

### 具体原因列表

| 原因 | 展开 |
|------|------|
| **工业界不用** | Claude Code、Cursor、Aider、OpenHands——所有工业级 Coding Agent 都自实现，无一用 LangChain |
| **循环太简单** | ReAct 循环去掉注释不到 30 行，不需要框架帮你写 |
| **真正的难点框架帮不了** | 5 层上下文管道、安全确认、流式双向通信——都是业务强相关的工程问题 |
| **抽象过重** | 一个简单 LLM 调用要经过 5-6 层 wrapper，出错时堆栈看不懂 |
| **JS 版二等公民** | LangChain.js 功能滞后于 Python 版，社区小 |
| **依赖膨胀** | 几十个子包，核心只需要 3 个依赖 |
| **控制权** | 用框架 = 用控制权换速度。AI 时代自实现速度已经很快，不需要这个交易 |

**LangChain 的 Memory vs Mako 的 5 层管道**：

| | LangChain Memory | Mako 5 层管道 |
|--|-----------------|--------------|
| BufferMemory | 全部保留（爆了就截断） | L1-L3 精细管理 |
| SummaryMemory | 调 LLM 摘要（一层） | L4 是 9 维结构化摘要（更精细） |
| VectorStoreMemory | 向量检索 | L5 用 BM25（轻量，够用） |
| 粒度 | 粗（整段对话） | 细（区分 tool/assistant/user，按轮次处理） |

## 叙述参考

**被问"你了解 RAG 吗"**：
> "RAG 本质是动态扩充上下文——先检索相关文档，塞进 prompt，让 LLM 有据可依。流程是：文档切 Chunk → Embedding 变向量 → 存向量库 → 查询时把问题也 Embed → 找最相似的 Top-K → 拼接到 prompt。Mako 没有用向量 RAG，因为编码上下文是有序的对话历史，不是散落的知识片段，BM25 关键词检索对代码场景够用。"

**被问"为什么不用 LangChain"**：
> "两个事实：一是所有工业级 Coding Agent 都不用——Claude Code、Cursor、Aider 全部自实现；二是 Agent 循环本身不到 30 行代码，框架省不了多少。真正的难点是上下文精细管理、安全确认、流式双向通信——这些需要精细控制每一个 token、每一个分支逻辑，经过一层框架抽象后很难做到，受框架约束反而更难实现。自实现的好处是极致轻量（3 个依赖）、完全可控、每个决策都能讲清楚 why。"

**被追问"那什么场景该用 LangChain / RAG"**：
> "LangChain 适合快速原型验证（10 分钟跑通一个 demo）和通用场景（客服机器人、文档问答）。RAG 适合从海量静态文档中检索知识（企业知识库、产品文档问答）。但 Coding Agent 需要的是精细控制对话历史，不是检索静态文档，所以场景不匹配。"
