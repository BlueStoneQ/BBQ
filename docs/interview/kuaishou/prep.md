## 1. H5/RN/Web 性能优化（主场）

> JD 第 2 条核心。你有完整的数据支撑，这是最强的牌。

### 你的数据

| 项目 | 指标 | 优化前 → 优化后 | 手段 |
|------|------|----------------|------|
| MT 优选 | 秒开率 | 10% → 78% | 数据预加载 + 合并 setData + 非渲染数据迁移 |
| XM 快应用 | 内存 PSS | 41MB → 35.8MB | DEX 布局优化，热代码前置减少 page fault |
| XM 快应用 | 包体 | 153MB → 60MB | 模块裁剪 + 条件编译 + 反射解耦 |
| XM 负载平台 | 上传耗时 | 121s → 42s（4.4G） | S3 MPU 分片并发 |

### 三层优化体系（RN 场景）

```
React 层（业务代码优化）
├── memo + useCallback + selector → 减少重渲染
├── FlashList 虚拟列表 → 列表性能
├── 状态下沉 → 避免全局 re-render
└── 懒加载 + 代码分割 → 首屏减负

RN 框架层（容器优化）
├── 分 Bundle（Common + Business）→ 首屏只加载必要代码
├── 容器池预热 → 引擎初始化提前
├── Hermes 预编译 → JS 解析 -50~70%
└── Reanimated 3 → 动画跑 UI 线程

Native 层（系统优化）
├── DEX 布局优化 → 启动内存降低
├── Native 预请求 → 网络不等 JS
├── ABI Split → 包体 -40~60%
└── R8 优化 → DEX -20~40%
```

### 话术

> "性能优化我按三层讲：React 层减重渲染（memo/selector/虚拟列表），RN 框架层减启动负担（分 Bundle/Hermes/容器预热），Native 层减系统开销（DEX 优化/预请求/包体裁剪）。每层都有具体数据——秒开率 10%→78%，内存 41→35.8MB，包体 153→60MB。"

### 可能追问

**Q: 秒开率 10%→78% 具体怎么做的？**
> 三个手段叠加：
> 1）数据预加载——路由跳转前就发请求（发布-订阅模式），页面 onLoad 时数据已经回来了
> 2）合并 setData——原来 5 次 setData 合并为 1 次，减少渲染次数
> 3）非渲染数据迁移——不需要渲染的数据从 data 移到普通变量，减少 diff 计算量

**Q: 怎么定义"秒开"？怎么测量？**
> 秒开 = 从用户点击到页面可交互 < 1s。测量用探针 SDK——有限状态机 + AOP 拦截页面生命周期，自动注入测速点，上报到监控平台。

## 4. 支付业务场景

> JD 加分项："支付/金融/信贷系统经验优先"。你在 XT 做过金融产品。

### 支付前端核心关注点

| 关注点 | 说明 | 你的相关经验 |
|--------|------|-------------|
| **金额精度** | 前端不用浮点数算钱，用分为单位（整数）或 BigDecimal | 金融产品开发 |
| **幂等性** | 防重复提交（按钮 loading + 请求去重 + 后端幂等 key） | 通用工程实践 |
| **安全** | HTTPS + 签名验证 + 敏感信息脱敏 + CSP | 通用安全意识 |
| **异常处理** | 支付超时/失败/部分成功的 UI 状态管理 | 弹窗治理（状态机思维） |
| **合规** | 实名认证、风控拦截、限额提示 | 一键注册（获取手机号） |

### 弹窗治理 → 支付场景映射

你在 XT 做的弹窗治理（四层架构）本质上就是支付场景的复杂交互管理：

```
支付场景的弹窗/交互冲突：
├── 风控拦截弹窗（最高优先级）
├── 实名认证弹窗
├── 支付确认弹窗
├── 营销优惠弹窗
└── 系统通知弹窗

你的方案：
├── UI 层：统一弹窗组件
├── 拦截层：AOP 拦截器
├── 调度层：优先级异步任务队列
└── 配置层：线上配置中心
```

### 话术

> "在 XT 做过金融产品的前端架构。最有代表性的是弹窗治理——支付/金融场景下弹窗冲突严重（风控拦截、实名认证、支付确认、营销优惠同时触发），我设计了四层架构：UI 层统一组件、拦截层 AOP 拦截、调度层优先级队列、配置层线上热配。本质是用状态机 + 优先级队列解决复杂交互的时序问题。"

## 6. 前端基础 & React 深度

> 基础要求，不能丢分。

### React 高频

| 考点 | 一句话 |
|------|--------|
| Fiber | 可中断渲染，链表遍历，支持时间切片和优先级调度 |
| Hooks 原理 | 链表存储，按顺序读取，Fiber 节点上挂 hook 对象 |
| useMemo vs useCallback | useMemo 缓存值，useCallback 缓存函数引用 |
| useEffect 清理 | return 函数在下次 effect 执行前或卸载时调用 |
| Concurrent Mode | 渲染可中断，startTransition 标记低优先级更新 |
| Zustand | 闭包 + 发布订阅 + useSyncExternalStore + selector 精准更新 |

### JS 高频

| 考点 | 核心 |
|------|------|
| 事件循环 | 同步 → 微任务（Promise.then）→ 宏任务（setTimeout） |
| 闭包 | 函数 + 词法环境引用，变量生命周期延长 |
| 原型链 | `__proto__` 链接形成链，属性查找沿链向上 |
| Promise | 状态容器（pending/fulfilled/rejected），链式调用 |
| this | 调用时决定，箭头函数继承外层 |

### HTTP

| 考点 | 核心 |
|------|------|
| 缓存 | 强缓存（Cache-Control）→ 协商缓存（ETag/Last-Modified） |
| CORS | 同源策略 + 预检请求（OPTIONS）+ Access-Control-Allow-Origin |
| HTTPS | HTTP + TLS，非对称交换密钥 → 对称加密传输 |
| HTTP/2 | 多路复用 + 头部压缩 + 服务端推送 |
