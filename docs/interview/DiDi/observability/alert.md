# 告警体系 + 发布观测 SOP

> 第一性原理：告警的本质 = **缩短"出问题"到"有人处理"的时间**。
>
> 没有告警的可观测 = 有眼睛但没有神经系统 — 看见了但大脑不知道。
>
> 两个核心问题：
> 1. 什么时候该喊人？（告警阈值）
> 2. 喊了之后多快能修好？（响应 SOP）

## 告警分级策略

> 本质：不是所有问题都值得半夜打电话。分级 = 用**影响面 × 紧急度**决定响应强度。

| 级别 | 定义 | 响应时间 | 通知方式 | 动作 |
|------|------|---------|---------|------|
| **P0** | 服务不可用 / 核心功能全损 | < 1min | 电话 + 群 @all | 立即回滚，先止血再排查 |
| **P1** | 性能严重退化 / 局部功能异常 | < 5min | 飞书/钉钉告警 + 值班 | 暂停发布，定位修复 |
| **P2** | 体验退化 / 非核心异常 | 当日 | 工单 + 排期 | 纳入 sprint 优化 |

**核心原则**：P0 不排查，先回滚。P1 不扩大，先止损。P2 不紧急，排期治理。

### 事后复盘（P0/P1 必须）

> 本质：止血只是第一步，复盘才能防止同类问题再发。P2 不需要复盘，排期治理即可。

**5 Why 复盘法**（MT 实践）：

```
案例：P95 响应时间从 200ms 飙到 3s

Why 1: 为什么 P95 飙了？→ 某接口响应 > 5s
Why 2: 为什么该接口慢？→ 数据库查询超时
Why 3: 为什么查询超时？→ 缺少索引，全表扫描
Why 4: 为什么缺索引？→ 新表上线未做 SQL Review
Why 5: 为什么没 Review？→ 没有 CI 卡控慢 SQL

根因：缺少 SQL 性能卡控机制
改进：CI 加入 explain 检测，新增表必须有索引覆盖
```

**复盘产出**：
| 项 | 内容 |
|---|------|
| 时间线 | 发现时间 → 响应时间 → 恢复时间（算 MTTR） |
| 根因 | 5 Why 推导到根本原因 |
| 止血动作 | 当时做了什么 |
| 改进项 | 防止再发的机制/工具/卡控 |
| 责任人 + deadline | 改进项必须有 owner 和 DDL |

**关键**：复盘不是追责，是**建机制**。每次复盘的改进项 = 系统韧性 +1。

## 阈值怎么定

> 第一性原理：阈值不是拍脑袋，是**数据驱动 + 业务约束 + 迭代调优**三者结合。

### 阈值的三个来源

| 层次 | 来源 | 适用场景 | 例子 |
|------|------|---------|------|
| **业界基线** | Google/SRE 公开标准 | 初始阶段、不知道怎么定 | LCP < 2.5s, 5xx < 0.1% |
| **历史 Baseline** | 自己采集 2-4 周数据 | 有运行数据后 | P95 正常 200ms → 告警设 800ms |
| **业务 SLA 倒推** | 和产品/业务约定的承诺 | 成熟阶段 | 99.9% 可用 → P0 MTTR < 5min |

### 作为 Leader 怎么定阈值

```
Step 1: 先裸跑采集 2 周（不设告警，只收数据）
Step 2: 拿到 baseline（正常情况各指标的 P50/P95/P99）
Step 3: 参考业界标准 + 自身 baseline → 定初版阈值
        告警阈值 ≈ baseline P95 × 2~3 倍
Step 4: 试运行 1-2 周 → 统计误报率/漏报率 → 调整
Step 5: 稳定后写入 SLA 文档，团队达成共识
```

**核心原则**：阈值不是一次定死的，是 baseline + 迭代调优。定太松 = 漏报，定太紧 = 告警风暴。

## Node.js（BFF / SSR）告警

> 场景：千人级负载分析平台，Nuxt SSR + BFF 聚合层，日常 QPS 800+，高峰 2000+。

| 指标 | 正常值 | P1 阈值 | P0 阈值 | 根因方向 |
|------|--------|---------|---------|---------|
| **P95 响应时间** | 200ms | > 800ms | > 3s | 慢查询 / 缓存穿透 / 上游超时 |
| **5xx 错误率** | < 0.05% | > 0.3% | > 2% | 上游 5xx / 未 catch / OOM |
| **Event Loop Lag** | 8ms | > 80ms | > 300ms | CPU 密集计算 / 同步 IO |
| **未捕获异常** | 0-2/h | > 10/min | > 50/min | 新代码 bug / 边界未覆盖 |
| **内存（Heap）** | 280MB 稳定 | 持续上涨 30min | RSS > 800MB | 内存泄漏 / 大对象未释放 |

## Web 前端告警

> 场景：B 端数据分析平台 + 活动搭建页面，SPA + SSR 混合。

| 指标 | 正常值 | P1 阈值 | P0 阈值 | 根因方向 |
|------|--------|---------|---------|---------|
| **LCP** | 1.8s | > 4s | — | 大图未压缩 / SSR 降级 / CDN 异常 |
| **CLS** | 0.05 | > 0.25 | — | 动态内容无占位 / 字体闪烁 |
| **INP** | 80ms | > 500ms | — | 长任务阻塞 / 重渲染 |
| **JS Error 率** | < 0.1% | > 1% | > 5% | 新版本 bug / 第三方脚本 |
| **白屏率** | < 0.3% | > 2% | > 5% | JS 加载失败 / 致命错误阻断渲染 |

## 搭建页面告警

> 场景：LowCode 搭建的活动/营销页，配置驱动渲染，非开发者产出。

| 指标 | 正常值 | P1 阈值 | 根因方向 |
|------|--------|---------|---------|
| **渲染成功率** | > 99.5% | < 97% | 组件 props 异常 / JSON 配置错误 |
| **白屏率** | < 0.5% | > 3% | 容器组件崩溃 / 数据源超时 |
| **组件加载失败率** | < 0.1% | > 1% | CDN 异常 / 版本不兼容 |

## 发布后观测 SOP — 后端（Node）

> 本质：发布 = 引入变量。观测 = 验证变量无害。时间窗口从短到长，逐步确认稳定。

| 阶段 | 时间窗口 | 关注指标 | 为什么 | 异常反应 |
|------|---------|---------|--------|---------|
| **金丝雀** | 0-5min | 5xx 错误率 + 未捕获异常 | 最快暴露"坏包" | 任一飙升 → **立即回滚** |
| **灰度放量** | 5-30min | P95 响应时间 + EL Lag | 性能退化需要流量才暴露 | P95 > 800ms → 暂停放量 |
| **全量观察** | 30min-24h | 内存趋势 + 异常数趋势 | 泄漏/低频 bug 需时间积累 | 内存不回落 → 排查泄漏 |
| **稳定确认** | 1-7d | WoW 对比（本周 vs 上周） | 确认无长尾退化 | 无退化 → 关闭观察窗口 |

**实际操作**：
```
发布命令 → 金丝雀 1 Pod（5% 流量）→ 看 5min 大盘
  ↓ 正常
灰度 30%（3 Pod）→ 看 30min
  ↓ 正常  
全量 → 当天持续关注 → 次日复查内存/异常趋势 → 7d 后 WoW 确认
```

## 发布后观测 SOP — 前端（Web）

> 本质：前端发布 = CDN 刷新 + 用户逐步加载新版本。没有"金丝雀 Pod"概念，靠灰度开关 + 实时监控。

| 阶段 | 时间窗口 | 关注指标 | 为什么 | 异常反应 |
|------|---------|---------|--------|---------|
| **灰度 5%** | 0-10min | JS Error 率 + 白屏率 | 新代码致命错误最先暴露 | 白屏 > 1% → 回退 CDN 版本 |
| **灰度 30%** | 10-60min | LCP + CLS + INP | 体验退化需要样本量 | LCP 退化 > 50% → 暂停 |
| **全量** | 1-24h | JS Error 趋势 + 用户反馈 | 长尾设备/浏览器兼容问题 | 特定机型报错 → hotfix |
| **稳定确认** | 1-3d | Core Web Vitals WoW | 确认无退化 | 无问题 → 发布完成 |

**前端灰度实现**：
```
方式 1: CDN 按 Cookie 分流（header: x-gray=true）
方式 2: Feature Flag（代码里 if/else 切换组件版本）
方式 3: HTML 入口分版本（gray.html / index.html）
```

## 告警质量治理

> 本质：告警太多 = 没有告警（狼来了效应）。告警体系也需要治理。

| 问题 | 表现 | 治理手段 |
|------|------|---------|
| **告警风暴** | 一个故障触发 50 条告警 | 告警聚合 + 根因收敛 |
| **误报** | 正常波动触发告警 | 调阈值 / 用环比而非绝对值 |
| **漏报** | 出事了没告警 | 定期 review 告警覆盖率 |
| **无人响应** | 告警了但没人处理 | 值班制度 + 升级机制（5min 无响应 → 升级） |

**健康指标**：告警响应率 > 95%，MTTR < 10min（P0）/ < 30min（P1）。
