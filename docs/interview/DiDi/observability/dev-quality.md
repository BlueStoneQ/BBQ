# 研发质量 + 人效 + AI 开发可观测性

> 第一性原理：运行时可观测回答"系统跑得好不好"，研发可观测回答"团队交付得好不好、快不快、AI 用得对不对"。
>
> 本质：不可度量的就不可治理。把研发过程也变成可观测的，才能用数据驱动改进。

## 研发质量可观测

> 回答：团队交付的代码质量如何？缺陷在哪个环节引入、哪个环节逃逸？

### 核心指标

| 维度 | 指标 | 采集来源 | 正常 | 差（需治理） |
|------|------|---------|------|------------|
| **代码质量** | Lint 通过率 | CI Pipeline | > 98% | < 90% |
| **代码质量** | 圈复杂度 > 10 的函数占比 | SonarQube / ESLint | < 5% | > 15% |
| **测试质量** | 行覆盖率 | Jest / Vitest | > 70% | < 40% |
| **测试质量** | 缺陷逃逸率 | Bug Tracker | < 10% | > 30% |
| **交付质量** | 线上 Bug 数/迭代 | Bug Tracker | < 3 | > 10 |
| **交付质量** | 发布回滚率 | CI/CD 平台 | < 2% | > 10% |
| **流程质量** | CR 驳回率 | Git 平台 | 10-30% | > 50% 或 < 5% |
| **流程质量** | CI 成功率 | CI 平台 | > 90% | < 70% |

### 治理手段

```
采集：CI 产出报告（覆盖率/Lint/复杂度）→ 聚合到 Dashboard
告警：覆盖率低于阈值 → 阻断 MR 合入
治理：
  - Lint 卡控（增量检查，lint-staged）
  - CR 质量（必须 2 人 approve）
  - 测试门禁（新增代码必须有对应用例）
验证：每迭代 review 指标趋势
```

## 研发人效可观测

> 回答：团队快不快？时间花在哪里？哪些环节是瓶颈？

### 核心指标

| 维度 | 指标 | 含义 | 采集来源 |
|------|------|------|---------|
| **交付速度** | 需求交付周期（Lead Time） | 从需求提出到上线 | 项目管理工具 |
| **交付速度** | 开发周期（Cycle Time） | 从开始开发到代码合入 | Git 平台 |
| **交付频率** | 发布频率 | 每周/每天发几次 | CI/CD 平台 |
| **阻塞时间** | CR 等待时长 | 提 MR 到首次 Review | Git 平台 |
| **阻塞时间** | CI 耗时 | 从 push 到 pipeline 完成 | CI 平台 |
| **重复劳动** | 手动操作占比 | 能自动化但还在手动做的 | 团队调研 |

### DORA 四指标（业界标准）

| 指标 | 优秀 | 中等 | 差 |
|------|------|------|-----|
| 部署频率 | 每天多次 | 每周 | 每月 |
| 变更前置时间 | < 1 天 | 1 周 | > 1 月 |
| 故障恢复时间（MTTR） | < 1 小时 | < 1 天 | > 1 周 |
| 变更失败率 | < 5% | < 15% | > 30% |

### 治理手段

```
瓶颈识别：CR 等待 > 4h → 自动提醒 reviewer
工具爆破：CLI / 搭建系统 / 调试提效 → 消除重复劳动
自动化：CI/CD 全自动 → 消除手动部署等待
度量驱动：每迭代 review DORA 指标趋势
```

## AI Native 开发可观测

> 回答：AI 工具用得好不好？提效多少？质量影响如何？

### 核心指标

| 维度 | 指标 | 含义 | 采集来源 |
|------|------|------|---------|
| **采纳度** | AI 代码采纳率 | 生成的代码被保留的比例 | IDE 插件统计 |
| **覆盖度** | AI 辅助覆盖率 | 哪些环节用了 AI（编码/测试/CR/文档） | 工具链埋点 |
| **效率** | AI 辅助前后 Cycle Time 对比 | 提效百分比 | Git + 项目工具 |
| **质量** | AI 生成代码的缺陷率 vs 人工 | AI 代码是否引入更多 Bug | Bug Tracker 标记 |
| **安全** | AI 代码安全扫描通过率 | 有没有引入漏洞 | SAST 工具 |

### 治理手段

```
采集：IDE 插件上报 AI 交互数据（生成/采纳/拒绝/修改）
分析：按团队/个人/场景维度聚合
告警：采纳率过低 → 可能工具不好用，需优化 Prompt/配置
治理：
  - 采纳率低 → 优化 Prompt 模板 / 更换模型
  - 缺陷率高 → 加 AI 代码 Review 环节 / 加测试门禁
  - 覆盖度低 → 培训 + 流程嵌入（CI 自动触发 AI 测试生成）
```

## 三者的闭环关系

```
研发质量可观测 ← "交付的好不好"
        ↕ 互相影响
研发人效可观测 ← "交付的快不快"
        ↕ AI 是加速器
AI 开发可观测 ← "AI 帮了多少忙、有没有帮倒忙"
```

**面试一句话**：我的可观测体系不只看运行时（性能/稳定性），还覆盖研发过程——质量、人效、AI 使用效果，三个维度都有指标、有度量、有治理闭环。这就是"全链路治理"的完整含义。
