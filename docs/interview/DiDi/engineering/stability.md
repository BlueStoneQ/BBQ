# 稳定性建设

> 解决什么问题：线上出了问题怎么快速发现、快速定位、快速恢复，以及怎么防止问题上线。
>
> 本质：稳定性 = 防（不上线）+ 发现（秒级感知）+ 定位（快速归因）+ 恢复（分钟级止血）。
>
> 场景：DD 运营平台（高可用要求）、MT 搭建系统（产出的活动页线上稳定性）、秒开率探针 SDK（监控基础设施）。

## 稳定性的四个阶段

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│   防    │ →  │  发现   │ →  │  定位   │ →  │  恢复   │
│         │    │         │    │         │    │         │
│ 不让问题│    │ 问题上线│    │ 找到根因│    │ 快速止血│
│ 上线    │    │ 后秒级  │    │         │    │         │
│         │    │ 感知    │    │         │    │         │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
  灰度发布       错误监控       SourceMap      自动回滚
  Feature Flag   性能告警       链路追踪       降级开关
  自动化测试     白屏检测       用户录屏       兜底 UI
```

## 防：问题不上线

| 手段 | 怎么做 | 卡在哪 |
|------|--------|--------|
| **灰度发布** | 新版本 1% → 10% → 50% → 100%，观察指标 | 部署阶段 |
| **Feature Flag** | 新功能用开关控制，不改代码即可关闭 | 运行时 |
| **自动化回归** | P0 链路 E2E 测试跑在 CI 里 | 发布前 |
| **类型系统** | TypeScript 编译时拦截大量错误 | 编码时 |
| **Code Review** | 人工 + AI Review 双重把关 | 合并前 |

```
灰度策略（分级发布）：
  内部测试（dogfooding）→ 1% 外部用户 → 10% → 全量
  
  每级观察：
    - JS 错误率是否上升
    - 接口成功率是否下降
    - 性能指标是否劣化
    - 用户反馈是否异常
  
  异常 → 自动/手动回滚到上一版本
```

## 发现：秒级感知

```
监控三件套：

  1. JS 错误监控
     window.addEventListener('error', handler)          // 同步错误 + 资源加载错误
     window.addEventListener('unhandledrejection', handler) // Promise 异常
     → 采集：错误信息 + 堆栈 + 用户信息 + 页面 URL
     → 上报：Sentry / 自研平台
     → 告警：错误率超阈值 → 自动通知 oncall

  2. 接口监控
     拦截 fetch/XMLHttpRequest
     → 采集：URL + 状态码 + 耗时 + 业务错误码
     → 告警：成功率 < 99% 或 P99 延迟 > 3s

  3. 白屏检测
     页面加载完成后检测根节点是否有内容
     → 采点：document.querySelector('#root').children.length === 0
     → 或用 MutationObserver 监听 DOM 变化超时
     → 白屏 → 上报 + 尝试自动刷新
```

### 上报策略

```
正常打点：new Image().src = url（简单不跨域）
页面卸载时：navigator.sendBeacon（保证不丢）
大数据量：fetch POST（批量上报）

采样策略（避免打爆后端）：
  错误上报：100%（错误不能漏）
  性能上报：10-20% 采样
  行为上报：1-5% 采样
```

## 定位：快速归因

| 手段 | 解决什么 | 怎么用 |
|------|---------|--------|
| **Source Map** | 压缩代码还原为源码 | 构建时生成，上传到 Sentry，不部署到线上 |
| **链路追踪 traceId** | 前端错误关联后端日志 | 请求 header 带 traceId，前后端串联 |
| **用户操作录屏** | 复现用户操作路径 | rrweb 录制 DOM 变化，错误时上报最近 N 秒 |
| **面包屑 Breadcrumbs** | 错误发生前的操作序列 | Sentry 自动收集（点击/路由跳转/接口调用） |
| **环境信息** | 设备/浏览器/网络 | UA + navigator.connection |

```
理想的错误报告：

  错误：TypeError: Cannot read property 'name' of undefined
  位置：src/pages/Activity/List.tsx:42:15（Source Map 还原）
  影响：过去 1h 命中 523 次，影响 89 个用户
  操作序列：登录 → 点击活动列表 → 筛选"进行中" → 报错
  环境：Chrome 120, iOS 17, 4G
  traceId：abc123（可查后端日志）
  录屏：[播放]
```

## 恢复：分钟级止血

| 手段 | 时效 | 场景 |
|------|------|------|
| **自动回滚** | 秒级 | 灰度阶段错误率飙升 → CDN 切回旧版本 |
| **手动回滚** | 分钟级 | oncall 判断后一键回滚 |
| **降级开关** | 秒级 | 配置中心关闭出问题的模块，不需要发版 |
| **兜底 UI** | 即时 | 请求失败展示缓存数据或默认状态 |
| **热修复 Hotfix** | 小时级 | 紧急修复 + 跳过灰度直接全量（仅 P0） |

```
oncall 应急响应流程：

  告警触发（错误率 > 1%）
  → oncall 收到通知（1min 内响应）
  → 判断严重程度：
      P0（核心链路挂了）→ 先回滚止血，再排查
      P1（部分功能异常）→ 降级开关关闭该模块
      P2（非核心问题）→ 记录 Bug，正常修复
  → 复盘：根因分析 + 防护措施 + 补充监控
```

## 运行时容错设计

```
前端容错的核心原则：局部挂不影响全局。

React ErrorBoundary（组件级隔离）：
  <ErrorBoundary fallback={<FallbackUI />}>
    <RiskyComponent />
  </ErrorBoundary>
  → RiskyComponent 报错 → 只显示 FallbackUI，其他区域正常

接口容错（降级策略）：
  try {
    data = await fetchAPI();
  } catch {
    data = getCachedData() || getDefaultData();  // 缓存兜底 / 默认值
  }

非核心模块容错：
  try {
    initAnalytics();   // 埋点 SDK 挂了不影响业务
    initAds();         // 广告模块挂了不影响核心功能
  } catch (e) {
    report(e);         // 静默上报，不阻断
  }
```

## 稳定性度量指标

| 指标 | 定义 | 目标 |
|------|------|------|
| **JS 错误率** | 有错误的 PV / 总 PV | < 0.1% |
| **接口成功率** | 成功请求 / 总请求 | > 99.9% |
| **白屏率** | 白屏 PV / 总 PV | < 0.01% |
| **MTTR** (Mean Time To Recovery) | 从发现到恢复的平均时间 | < 10min |
| **MTTD** (Mean Time To Detect) | 从问题出现到被发现的时间 | < 1min |
| **灰度拦截率** | 灰度阶段拦截的问题占总问题的比例 | > 80% |

## Q&A

**Q：前端稳定性建设你怎么做的？**

A：四个阶段——防（灰度 + 自动化测试 + 类型系统）、发现（错误监控 + 白屏检测 + 接口成功率告警）、定位（Source Map + traceId + 用户录屏）、恢复（自动回滚 + 降级开关 + 兜底 UI）。不是零散的点，是一套闭环体系。

**Q：白屏怎么检测？**

A：页面加载后定时检测根节点是否有子元素、或用 MutationObserver 监听 DOM 挂载超时。检测到白屏后自动上报 + 尝试刷新（最多一次，防死循环）。

**Q：错误监控和性能监控的上报方式为什么不同？**

A：错误上报 100% 不能漏（出了问题必须知道）；性能上报可以采样（统计意义上准确即可，全量上报打爆后端）。上报通道：正常用 Image beacon，页面关闭时用 sendBeacon 保证不丢。

**Q：灰度发布具体怎么实现的？**

A：CDN 层面按用户 ID hash 分流（或按 Cookie 标记灰度组）。命中灰度的用户请求到新版本资源，其余用户还是老版本。观察指标 OK 后扩大灰度比例直到全量。异常时 CDN 切回老版本即回滚。

**Q：ErrorBoundary 能捕获所有错误吗？**

A：不能。只捕获渲染阶段和生命周期中的同步错误。事件处理器里的错误、异步错误（Promise）、服务端渲染错误都捕获不到，需要配合 window.onerror 和 unhandledrejection。
