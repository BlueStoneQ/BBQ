```

## 1. AI 研发体系（最重要）

> JD 第 3 条明确写了"规划并落地 AI 开发体系"，这是你和其他候选人的最大差异。

### 你做了什么

```
AI 研发全链路（基于 MCP 协议）：

飞书 MCP（读需求/任务）
  → Figma MCP（读设计稿，生成代码）
  → Kiro/Agent（编码执行）
  → DevTools MCP（验证渲染结果）
  → Gerrit MCP（提交 Code Review）
  → Jira MCP（更新 Bug 状态）

效果：一人 Team 级规格全栈全链路交付
```

### 三层讲法

| 层级 | 内容 | 对应 JD |
|------|------|---------|
| **工具层** | AI Coding 工具深度使用（Copilot/Kiro/Cursor） | "深度使用 AI Coding 工具" |
| **框架层** | Mako Agent 框架（ReAct + Tool Use + 上下文管理） | "大模型应用和 Agent 工程落地" |
| **流程层** | MCP 串联全链路，AI 融入每个研发环节 | "AI 与研发流程的深度融合" |
| **业务层** | 运营工具 Agent 化——业务能力封装为 MCP Server，AI 直接操作业务 | "推动业务标准化改造" + Agent 落地 |

### 业务 Agent 化（核心差异化）

> 这是你和其他候选人拉开差距的关键点。别人讲"AI 辅助写代码"，你讲"AI 直接执行业务动作"。

```
传统运营工具链：
  运营人员 → 登录后台 → 手动配置活动/规则/投放 → 等开发联调 → 上线

Agent 化之后：
  AI Agent → 调用 MCP Server → 自动创建活动 / 配置规则 / 查询数据 → 即时生效

本质转变：
  从"人操作界面"→"AI 操作 API"
  从"搭建系统服务运营人员"→"MCP Server 服务 AI Agent"
  从"工具给人用"→"工具给 AI 用，人只做决策"
```

**怎么落地（四步走）**：

| 步骤 | 动作 | 产出 |
|------|------|------|
| 1. 能力标准化 | 运营动作抽象为原子操作（创建活动、配置规则、圈选人群...） | 业务能力清单 |
| 2. 协议化 | 每个原子操作封装为 MCP Tool（定义输入/输出 Schema） | MCP Server |
| 3. Agent 编排 | AI Agent 根据自然语言指令，选择合适的 Tool 组合执行 | 运营 Agent |
| 4. 人机协同 | 高风险操作（上线/大额投放）走人工审批，低风险自动执行 | 审批 + 自动化 |

**话术**：

> "我做 AI 体系不只停留在'辅助开发'这一层。更进一步的是——把运营业务能力封装为 MCP Server，让 AI Agent 能直接执行业务动作。比如运营说'创建一个针对北京新用户的 7 折优惠活动'，Agent 自动拆解为：圈选人群 → 配置规则 → 创建活动 → 提交审批。这不是 demo，是业务系统从'给人用'变成'给 AI 用'的架构升级。"

**可能追问**：

**Q: 这和直接调后端 API 有什么区别？**
> MCP 是标准协议（类比 USB），让任何 AI Agent 都能发现和调用业务能力，不需要为每个 Agent 单独写对接代码。而且 MCP 内置了能力描述（Schema），Agent 能自主决定用哪个工具、怎么组合。

**Q: 怎么保证 AI 不乱操作？**
> 分级策略：读操作自动执行 → 低风险写操作 Agent 自主执行 → 高风险操作（上线/资金）必须人工审批。Harness 层做兜底校验。

### 话术

> "我做 AI 研发体系分三层：第一层是工具引入——让团队用上 AI Coding 工具，这是最基础的；第二层是框架建设——我从零实现了 Mako Agent 框架，理解 Agent 的核心机制（ReAct 循环、Tool Use、上下文管理）；第三层是流程融合——基于 MCP 协议把需求、设计、编码、验证、Review、Bug 管理全部串起来，AI 不只是写代码的工具，而是贯穿研发全流程的协作者。"

### 可能追问

**Q: 怎么评估 AI 工具的效果？怎么量化？**
> 维度：代码产出速度（行/天）、Bug 率变化、Review 通过率、需求交付周期。不追求"AI 写了多少代码"，追求"端到端交付效率"。

**Q: AI 生成代码的质量怎么保障？**
> 三层保障体系：
> 1）**Steering（规范层）**——项目规范（编码风格、架构约定、技术栈约束）自动注入 AI 上下文，AI 遵守团队标准
> 2）**SDD / Spec-Driven Development（流程层）**——需求结构化拆解为 spec（需求→设计→任务），AI 按步骤实现而不是一口气写完，每步可 Review
> 3）**Harness（保障层）**——执行结果校验 + 全链路卡控（Lint → Hooks → CI/CD 对 AI 和人一视同仁），做错了自动回退
>
> 一句话：Steering 管"怎么写"，SDD 管"做什么"，Harness 管"做错了怎么办"。三层合在一起 = AI 质量保障体系。

**Q: 怎么推动团队接受 AI 工具？**
> 不强推。先自己用出效果 → 分享具体案例和数据 → 提供低门槛入口（配好环境、写好 prompt 模板）→ 让愿意尝试的人先跑起来 → 效果说话。

**Q: 怎么推动多个业务方向采纳你的工具？**

> 2）沉淀文档 + 使用指南
> 3）主动找其他团队 demo 演示
> 4）降低接入成本（一行命令搞定）
> 5）持续迭代，响应反馈

## 5. React + 工程化深度

### React 高频考点

| 考点 | 一句话 |
|------|--------|
| Fiber | 可中断的渲染单元，链表遍历替代递归，支持时间切片 |
| Concurrent Mode | 渲染可中断 + 优先级调度（Lane），用户输入优先于数据更新 |
| Hooks 原理 | 链表存储，按顺序读取，不能条件调用 |
| memo/useMemo/useCallback | 减少不必要的重渲染和重计算 |
| Server Components | 组件在服务端渲染，不发 JS 到客户端，减少 bundle |

### 工程化深度

| 方向 | 你的经验 |
|------|---------|
| Webpack/Vite | 构建配置、性能优化、插件开发 |
| CI/CD | 流水线设计、自动化发布、灰度策略 |
| 代码质量 | ESLint 规则集 + Git Hooks + CI 卡控 |
| 自动化测试 | pytest + uiautomator2（设备自动化） |
| 脚手架 | CLI + plugin 机制 + 项目模板 |

## 6. 运营增长业务理解

> 运营平台的核心诉求：快速上线活动、数据驱动决策、多业务复用。

### 运营平台典型架构

```
运营平台
├── 活动搭建（低代码/模板化）
├── 规则引擎（优惠券/红包/满减规则配置）
├── 人群圈选（用户分群 + 定向投放）
├── A/B 测试（灰度 + 数据对比）
├── 数据看板（实时指标 + 漏斗分析）

### 你的相关经验

- 灰度发布 = A/B 测试的技术基础

### 话术

> "运营平台的核心是'快'——快速上线活动、快速验证效果、快速迭代。我在 MT 做的搭建系统就是解决这个问题——把活动页面从开发交付变成运营自助搭建，活动创建从 0.5 人天降到 15 秒。"
