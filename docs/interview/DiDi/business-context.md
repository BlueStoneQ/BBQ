## 运营平台是什么

```
用户：运营人员（非技术人员）
场景：管理网约车业务的日常运营动作
目标：降低运营对开发的依赖，自助完成配置/发布/分析

不是做 C 端打车 App，是做运营人员用的 B 端管理平台（Web 后台）。
```

---

## 核心业务概念

| 概念 | 是什么 | 例子 |
|------|--------|------|
| **运营活动** | 优惠券/红包/补贴/满减活动 | "新用户首单立减 10 元" |
| **策略配置** | 针对不同条件的差异化规则 | "北京 8-10 点高峰期补贴 +5 元" |
| **灰度/ABTest** | 活动只对部分用户生效，看数据再全量 | "10% 用户看到新版优惠券样式" |
| **圈人/人群包** | 选一批目标用户推活动 | "30 天未下单的沉默用户" |
| **增长** | 拉新/促活/留存/召回 | "老带新分享得 20 元" |
| **动态定价** | 根据供需/时段/天气调整价格 | "暴雨高峰 1.5 倍" |
| **数据看板** | 业务指标可视化 | GMV/完单量/补贴花费/ROI |
| **业务标准化** | 把重复运营动作沉淀为可复用模板 | "节日活动模板，改改文案就能上" |
| **审批流** | 活动/策略上线需要审批 | 运营配好 → 主管审批 → 上线 |

## 叙述参考

### 被问"你了解运营业务吗"

> "我在 MT 打车做过运营方向的技术建设——本质是一样的：让运营人员自助配置活动和策略，降低对开发的依赖。我做过活动搭建系统，创建活动从 0.5pd 压缩到 15s。也做过策略配置的复杂表单（弹窗触发规则）。技术上就是搭建系统 + 规则引擎 + 微前端聚合。"

### 被问"你对网约车业务怎么理解"

> "运营平台的核心是'配置化'——活动/定价/策略/人群，都是规则 + 条件的排列组合。前端的价值是把这些复杂规则变成运营能看懂、能自己配的界面。同时通过标准化沉淀，让重复的运营动作变成模板，不用每次从零开发。"

### 被问"你入职后怎么上手这个业务"

> "运营平台的技术挑战和业务域无关——不管是打车还是外卖还是电商，运营平台的技术需求都是搭建 + 规则 + 看板 + 微前端。我有这套方法论，切换业务域的成本很低。"

## AI + 运营平台：业务 Chat 化构想

🔥🔥🔥

> DD JD 第三条："规划并落地 AI 开发体系，探索基于业务建模的代码生成"。
> 这不只是 AI 辅助写代码，更是 AI 辅助运营操作业务。

### 💡 核心思路

**把运营平台的表单配置入口，升级为 AI Chat 入口**——运营用自然语言描述需求，AI Agent 自动完成配置。

```
传统：运营手动填 20 个表单字段 → 预览 → 提交审批 → 上线
AI 化：运营在 Chat 里说"下周一北京新用户首单立减 8 元" → Agent 自动生成配置 → 人工确认 → 上线

= 从"人操作系统"变成"人指挥 AI 操作系统"
```

### 🏗️ 落地路径

```
两条路（视后端配合度选择）：

路径 A：推动后端开发 MCP Server
  后端团队把活动/策略/人群等业务 API 封装为 MCP Tool
  → 我们的 Agent 直接调用

路径 B：前端用 Node.js 封装 MCP 适配层（更现实）
  后端已有 REST API → 我们用 Node BFF 包一层 MCP Server
  → 不依赖后端改造，前端自己搞定
```

### 🔄 完整流程

```
运营同学 → Chat 入口（自然语言）
  "给 30 天未下单的北京用户发一张 10 元券，限本周五前使用"
       ↓
🤖 AI Agent（LLM 解析意图 + 提取参数）
       ↓
📋 生成规则说明（给运营看的人话）：
   "活动：10 元优惠券
    人群：北京 + 30 天未下单
    有效期：即日起至本周五
    预估覆盖：约 12 万人"
       ↓
👀 运营确认/微调 → ✅ 确认
       ↓
🔧 Agent 调用业务 MCP Tools：
   1. call create_coupon({ amount: 10, expire: '2026-06-13' })
   2. call create_audience({ city: 'beijing', inactive_days: 30 })
   3. call bindCouponToAudience({ coupon_id, audience_id })
   4. call submit_for_approval({ activity_id })
       ↓
✅ 进入审批流 → 主管审批 → 上线
```

### 🛠️ 前端具体做什么

| 模块 | 工作 |
|------|------|
| 💬 Chat UI | 对话组件 + Streaming 渲染 + 消息历史 |
| 📋 规则说明渲染 | Agent 生成的配置 → 渲染为运营可读的卡片式说明 |
| ✏️ 可编辑预览 | 运营确认前可以微调参数（回退为表单模式编辑） |
| 🔌 MCP Server（Node.js） | 用 BFF 封装现有后端 API 为 MCP Tool |
| 🛡️ 安全兜底 | AI 生成的配置不直接上线，必须人工确认 + 审批流 |
| 📊 执行反馈 | Agent 执行结果实时展示（成功/失败/部分成功） |

### 🎯 为什么这个构想有说服力

1. ✅ **DD JD 直接写了**这个方向（AI + 业务融合）
2. ✅ 你有 **MCP Tool 封装的实际经验**（Mako/MCP 全链路，不是空想）
3. ✅ 降低运营门槛 = 降低人力成本 = 老板爱听
4. ✅ 落地路径清晰：Node.js BFF 封装 MCP，不依赖后端改造
5. ✅ 安全可控：AI 只生成建议，人工确认后才执行

### 📈 收益

| 维度 | 收益 |
|------|------|
| **运营效率** | 配置活动从填 20 个表单字段 → 一句话描述，效率 10x+ |
| **降低门槛** | 新运营不需要学复杂后台操作，说人话就行 |
| **减少出错** | AI 生成 + 人工确认，比手动填表单更不容易漏字段/配错规则 |
| **推动公司运营 AI 化** | 从一个方向打样 → 推广到全公司运营平台（影响力辐射） |
| **减少研发投入** | 运营不再提"帮我加个配置项"的需求，AI 直接完成 |
