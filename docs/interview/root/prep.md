### 职责 3：性能与质量保障

> 持续改进移动应用的性能、安全性、稳定性和可扩展性。主导启动速度、渲染效率、内存管理等关键性能指标的优化。

**三个故事（每个 2 分钟讲完）**：

| 故事 | 指标 | 手段 |
|------|------|------|
| 启动内存 | PSS 41MB → 35.8MB | DEX 布局优化（热代码前置减少 page fault） |

**IoT 场景补充**：

- 后台常驻 = 内存管控更严格

## 三、RN ↔ Native 通信（BLE 场景）

**我的角色**：设计 BLE Native Module 的架构，不是写蓝牙底层。

### 架构设计思路

```
JS 层（React）
├── useBLE() Hook（封装连接/断开/数据收发）
├── BLEContext（全局设备状态管理）
└── 设备列表/控制面板 UI

↕ TurboModule 接口

Native 层（Android/iOS）
├── BLEModule（TurboModule 实现）
│   ├── scan() → Promise<Device[]>
│   ├── connect(deviceId) → Promise<void>
│   ├── disconnect(deviceId) → void
│   ├── write(deviceId, data) → Promise<void>
│   └── 事件：onDeviceFound / onStateChange / onDataReceived
└── BLE 底层（Android 专家负责）
```

### 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 通信方式 | TurboModule（新架构） | 同步能力 + 类型安全 + 懒加载 |
| 事件回传 | NativeEventEmitter | BLE 是持续状态流，不是一次性请求 |
| 状态管理 | JS 侧状态机 | 连接状态（断开/连接中/已连接/错误）用状态机管理 |
| 错误处理 | 重连策略 + 超时 + 降级 | IoT 场景网络不稳定是常态 |

### 我能带来什么（vs 普通 RN 开发）

- **架构层面**：不只是"能调通"，而是设计可扩展的 Native Module 体系（多设备/多协议/插件化）
- **性能层面**：理解 Bridge 开销，知道哪些操作要放 Native 线程、哪些可以回 JS
- **稳定性层面**：重连策略、心跳检测、异常兜底，从框架层保障而不是业务层 hack
