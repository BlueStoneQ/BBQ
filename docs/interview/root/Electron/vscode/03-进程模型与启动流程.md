## 第一性原理

**为什么需要多进程？**

单进程的致命问题：
1. 插件代码崩溃 → 整个 IDE 挂掉
2. 重计算任务 → UI 卡死 (Main Thread 被占)
3. 安全性 → 插件可以随意访问 IDE 内部状态

**VSCode 的答案**：隔离 + 专注
- 每个进程做一件事
- 通过 IPC 协议通信
- 崩溃只影响自身

## 各进程详解

### Main Process (主进程)

```
源码入口: src/main.ts → src/vs/code/electron-main/main.ts → app.ts
```

职责：
- Electron 应用生命周期管理 (`app.on('ready')`)
- BrowserWindow 创建和管理
- 原生菜单、对话框、通知
- 自动更新 (electron-updater)
- IPC Hub: 所有进程间通信的中枢

关键类：
- `CodeApplication` — 应用入口，组织启动流程
- `CodeWindow` — 对 BrowserWindow 的封装
- `LifecycleMainService` — 管理应用/窗口生命周期

### Renderer Process (渲染进程)

```
源码入口: src/vs/code/electron-browser/workbench/workbench.ts
```

每个编辑器窗口一个渲染进程。职责：
- 整个 Workbench UI 的渲染
- Monaco Editor 的运行
- 用户交互处理

重要约束：
- **开启 sandbox** — 无法直接访问 Node.js API
- 通过 `preload.js` 暴露有限的 IPC 通道
- 需要 Node 能力时，通过 IPC 调用 Main Process

### Extension Host (插件宿主进程)

```
源码入口: src/vs/workbench/services/extensions/
```

独立的 Node.js 子进程，是插件运行的沙箱。

类型：
| 宿主类型 | 运行环境 | 适用场景 |
|---------|---------|---------|
| LocalProcess | Node.js 子进程 | 大多数 Desktop 插件 |
| LocalWebWorker | Web Worker | 轻量插件 (Web 兼容) |
| Remote | 远程服务器进程 | Remote SSH/Container |

关键设计：
- 插件只能通过 `vscode` API 与 IDE 交互
- API 调用通过 RPC 跨进程到 Renderer
- 崩溃后自动重启，不影响 UI

### Utility Process (工具进程)

```
源码: src/vs/platform/utilityProcess/
```

Electron 的 `utilityProcess.fork()` 创建。用于：
- Shared Process (共享进程): 多窗口共享的服务 (扩展下载、设置同步)
- File Watcher: 文件变更监听
- Pty Host: 终端进程管理

## 启动流程时序

```
[用户双击]
    │
    ▼
┌─ Main Process ──────────────────────────────────────────────┐
│  1. src/main.ts                                              │
│     → parseCLIArgs()                                         │
│     → configureCommandlineSwitchesSync()                     │
│     → app.setPath('userData', ...)                           │
│     → protocol.registerSchemesAsPrivileged(...)              │
│                                                              │
│  2. app.once('ready') → onReady()                            │
│     → resolveNlsConfiguration() (国际化)                      │
│     → bootstrapESM()                                         │
│     → import('vs/code/electron-main/main.js')                │
│                                                              │
│  3. vs/code/electron-main/main.ts                            │
│     → CodeMain.main()                                        │
│     → 创建 InstantiationService (主进程DI容器)                 │
│     → 注册核心服务 (lifecycle, log, state...)                  │
│     → 创建 CodeApplication                                    │
│                                                              │
│  4. CodeApplication.startup()                                │
│     → 创建 IPC Server (electron IPC)                          │
│     → 启动 SharedProcess                                      │
│     → 创建第一个 CodeWindow                                    │
│                                                              │
│  5. CodeWindow → new BrowserWindow()                         │
│     → loadURL('vs/code/electron-browser/workbench/...')      │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Renderer Process ──────────────────────────────────────────┐
│  6. workbench.html → preload.js → workbench.ts               │
│     → 创建 Workbench InstantiationService                     │
│     → 注册 Workbench 服务                                     │
│     → Workbench.startup()                                    │
│       → 创建 Layout (Grid系统)                                │
│       → 渲染 Parts (TitleBar, SideBar, Editor, Panel...)     │
│       → 激活 contrib 模块                                     │
│                                                              │
│  7. ExtensionService.startExtensionHosts()                   │
│     → fork Extension Host 子进程                              │
│     → 建立 RPC 通道                                           │
│     → 激活已安装的插件                                         │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Extension Host ────────────────────────────────────────────┐
│  8. extensionHostProcess.ts                                  │
│     → 建立与 Renderer 的 IPC 连接                             │
│     → 加载插件 manifest                                       │
│     → 按激活事件 (activationEvents) 触发插件 activate()        │
└──────────────────────────────────────────────────────────────┘
```

## 性能优化: 启动加速策略

| 策略 | 实现方式 | 收益 |
|------|---------|------|
| Code Cache | V8 code cache 缓存编译后的 JS | 减少 JS 解析时间 |
| 延迟加载 | `InstantiationType.Delayed` | 非关键服务按需创建 |
| ESM 按需导入 | 动态 `import()` | 不加载未用模块 |
| 恢复上次状态 | 序列化编辑器/面板状态 | 减少重建开销 |
| Preload | preload.js 提前暴露 IPC | 渲染进程无需等 Main 就绪 |
| 标记点 | `perf.mark()` 全链路打点 | 可度量可优化 |

## 源码中的 perf 标记点

```typescript
perf.mark('code/didStartMain');           // 主进程入口
perf.mark('code/mainAppReady');           // app.ready
perf.mark('code/willLoadMainBundle');     // 开始加载主 bundle
perf.mark('code/didRunMainBundle');       // 主 bundle 执行完毕
// ... Renderer 侧
perf.mark('code/willStartWorkbench');     // Workbench 开始
perf.mark('code/didStartWorkbench');      // Workbench 就绪
```

可通过 `Developer: Startup Performance` 命令查看完整启动耗时瀑布图。

## Sandbox 模式的影响

VSCode 现在默认启用 Chromium Sandbox：

```
Renderer Process:
  ├── 无 Node.js API ❌
  ├── 无 require() ❌
  ├── 无 fs/child_process ❌
  └── 只有 preload.js 暴露的有限接口 ✅
```

这意味着：
- 所有需要 Node 的操作必须通过 IPC 委托给 Main/Utility Process
- preload.js 只暴露 `ipcRenderer.invoke/send/on` 包装
- 提升安全性，但增加了 IPC 开销

## 小结

| 设计决策 | 第一性原理 |
|---------|-----------|
| 插件运行在独立进程 | 故障隔离 — 插件崩溃不死 IDE |
| Renderer 开启 Sandbox | 最小权限 — Web 渲染器不需要 fs 权限 |
| Main 只做协调不做计算 | 职责单一 — 保证窗口管理响应性 |
| Utility Process 做重活 | 专注 — 文件搜索/监听不阻塞 UI |
| 延迟启动 Extension Host | 首屏优先 — 用户先看到 UI 再加载插件 |
