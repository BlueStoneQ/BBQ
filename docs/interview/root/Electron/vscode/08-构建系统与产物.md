## 第一性原理

**构建的本质**：源码 → 可运行产物

复杂度来源：
1. 多目标平台 (Windows/macOS/Linux × x64/arm64)
2. 多产物形态 (Desktop / Web / Server)
3. 60 万行 TypeScript 需要编译 + tree-shaking + minify
4. 100+ 内置插件需要独立打包
5. 原生模块需要针对 Electron 版本 rebuild

## 构建工具链

```
┌─────────────────────────────────────────────┐
│  Gulp (任务编排)                              │
│  ├── TypeScript Compiler (tsc)              │
│  ├── esbuild / swc (快速编译)               │
│  ├── Webpack (插件打包)                      │

│      ├── NSIS / Inno Setup (Windows)        │
│      ├── dmg-builder (macOS)                │
│      └── dpkg / rpmbuild (Linux)            │
└─────────────────────────────────────────────┘
```

## 关键 npm scripts

从 `package.json` 中提取关键构建命令：

```json
{
  "compile": "gulp compile",                          // 开发编译
  "watch": "npm-run-all -lp watch-client watch-extensions",  // 开发监听
  "compile-build": "gulp compile-build-with-mangling", // 生产编译(含混淆)
  "minify-vscode": "gulp minify-vscode",              // 压缩核心代码
  "build:deb": "gulp vscode-linux-x64 && gulp vscode-linux-x64-build-deb",  // Linux deb
  "build:exe": "gulp vscode-win32-x64 && gulp vscode-win32-x64-user-setup", // Windows
  "build:mac": "gulp vscode-darwin-x64",              // macOS
  "build:mac_arm": "gulp vscode-darwin-arm64"         // macOS ARM
}
```

## 构建流程 (以 Linux deb 为例)

```
Step 1: 编译 TypeScript
  gulp compile-build-with-mangling
  ├── tsc 编译所有 .ts → .js
  ├── mangling: 将私有属性名压缩 (减少包体积)
  └── 输出到 out/ 目录

Step 2: 打包核心代码
  gulp minify-vscode
  ├── 将 out/ 中的模块打成 bundle
  ├── 移除 sourcemap (生产构建)
  └── tree-shaking 未使用的代码

Step 3: 编译内置插件
  gulp compile-extensions-build
  ├── 每个 extensions/* 独立编译
  ├── webpack 打包插件代码
  └── 复制 node_modules (平台相关原生模块)

Step 4: 组装 Electron
  gulp vscode-linux-x64
  ├── 下载 Electron 预编译二进制
  ├── 将编译产物放入 resources/app/
  ├── rebuild 原生 Node 模块 (node-pty, spdlog...)
  ├── 应用 product.json (品牌/配置)
  └── 输出到 .build/linux/x64/

Step 5: 打包安装包
  gulp vscode-linux-x64-build-deb
  ├── 创建 deb 目录结构
  ├── 写入 control 文件 (包名/版本/依赖)
  ├── 安装桌面文件 (.desktop)
  └── dpkg-deb 打包 → .deb 文件
```

## 原生模块处理

VSCode 使用的原生 Node.js 模块：

| 模块 | 用途 | 为什么需要原生 |
|------|------|---------------|
| `node-pty` | 终端伪终端 | 需要操作系统 PTY 接口 |
| `@vscode/spdlog` | 高性能日志 | C++ 的 spdlog 库 |
| `native-keymap` | 键盘布局检测 | 需要读取系统键盘映射 |
| `native-watchdog` | 进程看门狗 | 检测主进程假死 |
| `@parcel/watcher` | 文件监听 | 原生 FSEvents/inotify |
| `kerberos` | 企业认证 | Kerberos 协议栈 |
| `@vscode/sqlite3` | 数据库 | 存储工作区状态 |

这些模块需要：
1. 针对 Electron 的 Node.js 版本编译 (ABI 兼容)
2. 针对目标平台交叉编译 (Linux/macOS/Windows)

```bash
# electron-rebuild 自动处理
npx electron-rebuild -v 37.2.3  # Electron 版本
```

## Mangling (标识符混淆)

VSCode 独特的优化：将所有**私有成员**名压缩为短名：

```typescript
// 源码
class FileService {
  private _onDidChangeFile = new Emitter<IFileChange>();
  private _fileSystemProviders = new Map();
}

// Mangling 后
class FileService {
  private _a = new Emitter<IFileChange>();
  private _b = new Map();
}
```

收益：
- 减少 ~20% bundle 体积
- 混淆内部实现细节

约束：
- 仅混淆 `private` 成员 (下划线开头)
- 不混淆 public API
- 需要稳定的 mangling map (不能每次构建结果不同)

## Code Cache (V8 代码缓存)

```typescript
// src/main.ts
const codeCachePath = path.join(userDataPath, 'CachedData', commit);
```

V8 将编译后的字节码缓存到磁盘。下次启动直接加载字节码，跳过 JS 解析。

生效条件：
- 相同的 commit hash (版本更新后失效)
- 相同的 V8 版本

## 小结

| 设计决策 | 第一性原理 |
|---------|-----------|
| Gulp 编排 | 任务可组合 — 开发/生产/CI 用同一套流程 |
| Mangling | 信息论 — 私有名不含信息量，压缩无损 |
| Code Cache | 避免重复计算 — 相同代码只编译一次 |
| 原生模块 Rebuild | ABI 兼容 — Electron 和 Node.js 版本对齐 |
| 分平台打包 | 一次编译多目标 — 源码相同产物不同 |
| ESM 迁移 | 标准化 — 拥抱平台能力而非自建加载器 |
