# VS Code 多进程架构

> 解决什么问题：编辑器要同时做 UI 渲染 + 语法分析 + 插件执行 + 文件监听 + 搜索，如果全跑在一个进程里，任何一个卡住/崩溃都会导致整个编辑器无响应。

## 各进程职责

| 进程 | 数量 | 语言 | 职责 | 崩溃影响 |
|------|------|------|------|---------|
| **Main** | 1 | Node.js + Chromium | 窗口/菜单/生命周期/启动子进程 | 全部挂 |
| **Renderer** | 每窗口 1 | Web（TS + DOM） | 编辑器 UI / Monaco / 布局 | 当前窗口白屏 |
| **Extension Host** | 1（每窗口） | Node.js | 运行所有插件 | 插件功能丢失，可重启 |
| **Language Server** | 每语言 1 | 任意（Node/Go/Rust） | 语法分析/补全/诊断 | 该语言功能丢失，可重启 |
| **Terminal** | 每终端 1 | pty 进程 | Shell 执行 | 该终端关闭 |
| **File Watcher** | 1 | Node.js | 监听文件变化 | 文件变化无法检测 |
| **Search** | 按需 | ripgrep (Rust) | 全局文本搜索 | 搜索失败 |

## 进程间通信

```
VS Code 的进程通信不是全用 Electron IPC（Mojo），有多种方式：

  Main ↔ Renderer：Electron IPC（ipcMain/ipcRenderer）
  Renderer ↔ Extension Host：自定义 RPC 协议（基于 MessagePort 或 IPC）
  Extension Host ↔ Language Server：LSP 协议（基于 stdin/stdout 或 socket）
  Main ↔ 子进程：child_process 的 stdin/stdout 或 IPC channel

= 不同进程对之间用最合适的通信方式
= 统一封装为 RPC 调用风格（请求/响应 + 事件通知）
```

## 为什么这么设计

```
核心原则：隔离一切"可能出问题"的东西

  问题 1：插件质量参差不齐，可能有 bug 导致崩溃
  解法：插件跑在 Extension Host → 崩溃只影响插件，重启即可

  问题 2：Language Server 做语法分析可能很慢（TypeScript 大项目分析 10s+）
  解法：Language Server 独立进程 → 卡住不影响 UI 渲染

  问题 3：全局搜索扫描磁盘可能很慢
  解法：搜索用独立的 ripgrep 进程 → 不阻塞任何其他功能

  问题 4：终端执行用户命令可能产生大量输出
  解法：每个终端独立 pty 进程 → 不影响编辑器

= 每种"重型/不可控"的操作都隔离到独立进程
= 崩溃可恢复、卡顿不传染
```

## 和你的快应用 IDE 的对应

| VS Code | 快应用 IDE（你的经验） |
|---------|---------------------|
| Main Process | Main Process（窗口管理 + 启动子进程） |
| Renderer | Renderer（Monaco Editor + UI 面板） |
| Extension Host | Extension Host（复用 VS Code 插件机制） |
| Language Server | 快应用专属 Language Server（语法/补全） |
| 子进程：Search | 子进程：依赖分析引擎（遍历者+访问者模式） |
| 子进程：Terminal | 子进程：设备通信（adb/CDP 调试协议） |
| 子进程：Build | 子进程：构建编译（webpack/快应用编译器） |
