## 第一性原理

**IDE 插件系统的终极矛盾**：
- 想让插件能做任何事 (强大) 
- 又不想让插件搞垮 IDE (安全 + 稳定)

**VSCode 的核心设计哲学**：
1. **进程隔离** — 插件跑在独立进程，崩了不影响 UI
2. **API 单向暴露** — 插件只能通过 `vscode` API 与 IDE 交互，不能直接操作 DOM
3. **声明式贡献** — UI 扩展点通过 package.json 声明，不是代码注入
4. **懒激活** — 插件只在需要时才加载执行

## 插件的生命周期

```
[安装/发现]
    │
    ▼
[解析 package.json] → 提取 contributes + activationEvents
    │
    ▼
[注册贡献点] → 声明式注册: 命令、菜单、视图、语言...
    │
    ▼
[等待激活事件] → onLanguage:python, onCommand:xxx, * ...
    │
    ▼
[激活] → require(extension.main) → extension.activate(context)
    │
    ▼
[运行] → 插件通过 vscode.* API 与 IDE 交互
    │
    ▼
[停用] → extension.deactivate() → 清理资源
```

## 激活事件 (Activation Events)

```json
// package.json
{
  "activationEvents": [
    "onLanguage:javascript",        // 打开JS文件时
    "onCommand:myExtension.start",  // 执行命令时
    "onView:myCustomView",          // 视图可见时
    "onFileSystem:ftp",             // 访问ftp://时
    "onStartupFinished",            // IDE启动完成后
    "*"                             // 立即激活 (不推荐)
  ]
}
```

设计原则：**能不加载就不加载**。大多数插件只有用到时才激活。

## 贡献点 (Contribution Points)

插件通过 `package.json` 的 `contributes` 字段声明式扩展 IDE：

```json
{
  "contributes": {
    "commands": [...],           // 注册命令
    "menus": {...},             // 菜单项
    "views": {...},            // 侧边栏视图
    "viewsContainers": {...},  // 视图容器
    "languages": [...],        // 语言声明
    "grammars": [...],         // TextMate 语法
    "themes": [...],           // 主题
    "snippets": [...],         // 代码片段
    "configuration": {...},    // 配置项
    "keybindings": [...],      // 快捷键
    "debuggers": [...],        // 调试器
    "taskDefinitions": [...]   // 任务类型
  }
}
```

**关键洞察**：贡献点是**声明式**的——IDE 解析 JSON 就能渲染菜单和 UI，不需要执行插件代码。这是性能和安全的双赢。

## Extension Host 类型

| 类型 | 进程 | 能力 | 场景 |
|------|------|------|------|
| LocalProcess | Node.js 子进程 | 完整 Node API | 文件操作、调试、终端 |
| LocalWebWorker | Web Worker | 无 Node, 无 DOM | 轻量计算、语言分析 |
| Remote | 远程 Node.js | 远程文件系统 | SSH Remote / Container |

插件在 `package.json` 中声明兼容哪种运行环境：

```json
{
  "extensionKind": ["workspace"],  // 在工作区侧运行 (Remote优先)
  "browser": "dist/web/extension.js"  // Web Worker 入口
}
```

## API 桥接: ExtHost ↔ MainThread

以 `vscode.commands.registerCommand` 为例：

```
[插件代码]
  vscode.commands.registerCommand('hello', () => 'world')
      │
      ▼
[ExtHostCommands] (Extension Host 进程)
  → 本地注册命令到 Map
  → this._proxy.$registerCommand('hello')  // RPC 到 Renderer
      │
      ▼
[MainThreadCommands] (Renderer 进程)
  → CommandsRegistry.registerCommand('hello', ...)
  → 命令出现在 Command Palette 中

[用户触发命令]
      │
      ▼
[MainThreadCommands]
  → this._proxy.$executeCommand('hello')  // RPC 回 Extension Host
      │
      ▼
[ExtHostCommands]
  → 从 Map 中找到 handler
  → handler() → 'world'
  → 返回结果给 MainThread
```

## API 安全边界

插件**不能**做的事：
- ❌ 直接操作 DOM (没有 `document` 对象)
- ❌ 直接访问 IDE 内部服务
- ❌ 修改其他插件的状态
- ❌ 阻塞 UI 线程 (因为在另一个进程)

插件**能**做的事 (通过 API)：
- ✅ 读写文件 (`vscode.workspace.fs`)
- ✅ 操作编辑器 (`vscode.window`, `vscode.TextEditor`)
- ✅ 提供语言能力 (`vscode.languages`)
- ✅ 创建终端 (`vscode.window.createTerminal`)
- ✅ 展示 WebView (`vscode.window.createWebviewPanel`)
- ✅ 注册命令、菜单、视图

## 插件间通信

```typescript
// 方式1: 通过命令
vscode.commands.executeCommand('other.extension.command', arg);

// 方式2: 通过 Extension API 暴露
// 插件A的 activate() 返回值
export function activate() {
  return { myApi: { doSomething() { } } };
}

// 插件B获取
const extA = vscode.extensions.getExtension('publisher.extensionA');
const api = await extA.activate();
api.myApi.doSomething();
```

## Proposed API

新 API 先标记为 proposed，只有白名单插件能用：

```json
// product.json
{
  "extensionEnabledApiProposals": {
    "GitHub.copilot-chat": ["chatParticipantAdditions", "languageModelSystem", ...]
  }
}
```

这让 VSCode 能在正式发布前迭代 API 设计。

## 性能与隔离机制

| 机制 | 解决的问题 |
|------|-----------|
| 独立进程 | 插件崩溃不影响 UI |
| 懒激活 | 启动不加载不需要的插件 |
| 声明式贡献 | UI 渲染不需要执行插件代码 |
| API 沙箱 | 插件无法直接访问内部状态 |
| Extension Host 重启 | 插件宿主崩溃后自动恢复 |
| Web Worker Host | 完全隔离 Node 能力 |

## 小结

| 设计决策 | 第一性原理 |
|---------|-----------|
| 进程隔离 | 故障域最小化 — 一个插件不能杀死 IDE |
| API 单向暴露 | 最小权限 — 插件只能通过受控接口操作 |
| 声明式贡献 | 数据驱动 — 解析 JSON 比执行代码快且安全 |
| 懒激活 | 按需加载 — 100个插件不等于100倍启动开销 |
| Proposed API | 渐进稳定 — 先内测再公开，API 不轻易废弃 |
| 双向 RPC | 职责分离 — ExtHost 执行逻辑，MainThread 操作 UI |
