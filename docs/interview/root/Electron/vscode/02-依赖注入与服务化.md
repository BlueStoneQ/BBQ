## 第一性原理

**问题**：IDE 有 100+ 个服务（文件系统、配置、日志、窗口...），它们互相依赖。如果手动 `new`，依赖链爆炸。

**本质需求**：
1. 解耦创建与使用 — 使用者不关心实现类是什么
2. 生命周期管理 — 单例、按需创建、统一销毁
3. 可替换性 — 测试时换 Mock，不同环境换不同实现

**VSCode 的答案**：自研轻量 DI 容器（不用 Angular/InversifyJS），基于 TypeScript 装饰器 + 服务标识符。

## 核心概念

```typescript
// 1. 定义服务标识符 (ServiceIdentifier)
const IFileService = createDecorator<IFileService>('fileService');

// 2. 声明接口
interface IFileService {
  readFile(uri: URI): Promise<Uint8Array>;
  writeFile(uri: URI, content: Uint8Array): Promise<void>;
}

// 3. 实现类
class DiskFileService implements IFileService {
  constructor(
    @ILogService private readonly logService: ILogService  // ← 装饰器声明依赖
  ) {}
  
  async readFile(uri: URI) { /* ... */ }
  async writeFile(uri: URI, content: Uint8Array) { /* ... */ }
}

// 4. 注册
registerSingleton(IFileService, DiskFileService, InstantiationType.Delayed);
```

## 运行机制拆解

### Step 1: createDecorator — 创建服务标识

```typescript
// vs/platform/instantiation/common/instantiation.ts
export function createDecorator<T>(serviceId: string): ServiceIdentifier<T> {
  const id = function (target: any, key: string, index: number): void {
    // 将服务ID记录到类的元数据上
    storeServiceDependency(id, target, index);
  };
  id.toString = () => serviceId;
  return id as any;
}
```

关键点：返回值**既是装饰器，又是标识符**。一个对象两种用途。

### Step 2: 参数装饰器 — 声明依赖

```typescript
class EditorService {
  constructor(
    @IFileService private fileService: IFileService,      // 参数0
    @IConfigurationService private config: IConfigurationService  // 参数1
  ) {}
}
```

装饰器执行时，会在 `EditorService` 的元数据上记录：
- 参数 0 → 需要 `IFileService`
- 参数 1 → 需要 `IConfigurationService`

### Step 3: InstantiationService — 核心容器

```typescript
class InstantiationService {
  private _services: ServiceCollection;  // 已注册的服务/描述符
  
  // 核心方法: 创建实例
  createInstance<T>(ctor: new (...args: any[]) => T, ...args: any[]): T {
    // 1. 读取 ctor 的参数装饰器 → 得到依赖的服务列表
    const deps = getServiceDependencies(ctor);
    
    // 2. 递归解析每个依赖 (可能触发延迟实例化)
    const resolvedDeps = deps.map(dep => this._getOrCreateServiceInstance(dep.id));
    
    // 3. 合并用户传入的参数 + 解析的服务
    return new ctor(...args, ...resolvedDeps);
  }
  
  // 获取或创建服务实例
  private _getOrCreateServiceInstance(id: ServiceIdentifier<any>) {
    const thing = this._services.get(id);
    if (thing instanceof SyncDescriptor) {
      // 延迟实例化: 现在才真正创建
      return this._createAndCacheServiceInstance(id, thing);
    }
    return thing;  // 已经是实例了
  }
}
```

### Step 4: 依赖图解析

当服务 A 依赖 B，B 依赖 C 时：

```
InstantiationService._createAndCacheServiceInstance(A)
  → 发现 A 依赖 [B, C]
  → 构建依赖图
  → 拓扑排序: C → B → A
  → 按序创建
```

使用 `graph.ts` 中的有向图实现，检测循环依赖会抛错。

## 两种注册方式

### 方式一: 直接注册实例

```typescript
const services = new ServiceCollection();
services.set(ILogService, new ConsoleLogService());  // 直接传实例
```

### 方式二: 注册描述符 (延迟创建)

```typescript
services.set(IFileService, new SyncDescriptor(DiskFileService));
// 第一次被依赖时才会实例化 DiskFileService
```

### 方式三: 全局注册 (contrib 模式)

```typescript
// 在模块加载时自动注册 (副作用导入)
registerSingleton(IFileService, DiskFileService, InstantiationType.Delayed);
```

`workbench.desktop.main.ts` 通过 import 这些模块文件触发注册。

## 实例化时机

| InstantiationType | 含义 | 场景 |
|---|---|---|
| `Eager` | Workbench 启动时立即创建 | 生命周期关键服务 |
| `Delayed` | 首次被请求时创建 | 大多数服务 |

## 为什么不用现成框架？

| 对比 | Angular DI | InversifyJS | VSCode DI |
|------|-----------|-------------|-----------|
| 装饰器方式 | 类装饰器 `@Injectable()` | 类+参数 | 仅参数装饰器 |
| 容器 | 模块级 | 全局/模块 | 层级容器 (可继承) |
| 体积 | 重 | 中 | 极轻 (~500行) |
| 特性 | 完整生态 | 完整 | 按需最小化 |

VSCode 选择自研的原因：
1. **极致轻量** — 核心代码不到 500 行
2. **层级容器** — Main/Workbench 各有容器，可继承可覆盖
3. **与 TypeScript 装饰器完美匹配** — 不需要额外注解
4. **无反射运行时** — 编译期就确定依赖关系

## 层级容器模式

```
MainProcess InstantiationService
  ├── services: [ILifecycleService, IWindowService, ...]
  │
  └── child → Workbench InstantiationService
                ├── services: [IEditorService, IViewService, ...]
                │
                └── child → ExtensionHost InstantiationService
                              └── services: [IExtHostService, ...]
```

子容器找不到服务时，会向父容器查找。这实现了：
- 服务作用域隔离
- 不同生命周期的服务分层管理

## 实战: 如何新增一个服务

```typescript
// 1. src/vs/platform/myFeature/common/myFeature.ts
export const IMyFeatureService = createDecorator<IMyFeatureService>('myFeatureService');
export interface IMyFeatureService {
  readonly _serviceBrand: undefined;
  doSomething(): void;
}

// 2. src/vs/platform/myFeature/common/myFeatureService.ts
export class MyFeatureService implements IMyFeatureService {
  declare readonly _serviceBrand: undefined;
  
  constructor(
    @ILogService private readonly logService: ILogService
  ) {}
  
  doSomething() {
    this.logService.info('MyFeature did something');
  }
}

// 3. 注册 (在 workbench.desktop.main.ts 中 import)
registerSingleton(IMyFeatureService, MyFeatureService, InstantiationType.Delayed);

// 4. 使用 (任何地方通过构造函数注入)
class SomeContribution {
  constructor(@IMyFeatureService private myFeature: IMyFeatureService) {}
}
```

## 小结

| 设计决策 | 为什么 |
|---------|-------|
| 参数装饰器而非类装饰器 | 不侵入类定义，接口和实现分离 |
| ServiceIdentifier 是函数 | 一个对象同时作为 ID 和装饰器 |
| SyncDescriptor 延迟创建 | 避免启动时创建所有服务，提升性能 |
| 层级容器 | 匹配进程/生命周期分层 |
| `_serviceBrand` | TypeScript 类型系统的 nominal typing hack |
