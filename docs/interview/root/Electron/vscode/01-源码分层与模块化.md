## 第一性原理

**大型软件的本质矛盾**：代码量增长 → 模块间耦合指数上升 → 维护成本不可控

**VSCode 的解法**：双维度切割
- **纵向**: 按抽象层次分层（base → platform → editor → workbench）
- **横向**: 按运行环境分目录（common / browser / node / electron-main）

## 纵向分层: 源码层级

```
src/vs/
├── base/          ← 第0层: 工具库 (不依赖任何上层)
├── platform/      ← 第1层: 平台服务 (依赖 base)
├── editor/        ← 第2层: 编辑器内核 (依赖 base + platform)
├── workbench/     ← 第3层: IDE整体框架 (依赖所有下层)
├── code/          ← 入口层: 各环境启动入口
└── server/        ← Remote Server 入口
```

### 各层职责

| 层 | 路径 | 核心职责 | 依赖规则 |
|----|------|---------|---------|
| base | `vs/base/` | 数据结构、事件、生命周期、DOM操作、IPC基础设施 | 不依赖上层 |
| platform | `vs/platform/` | 文件系统、配置、日志、存储、扩展管理等抽象服务 | 仅依赖 base |
| editor | `vs/editor/` | Monaco 编辑器核心（文本模型、视图、语言服务） | 依赖 base + platform |
| workbench | `vs/workbench/` | IDE 壳子（面板、视图、服务注册、插件宿主） | 依赖所有下层 |

### 关键约束

```
workbench → editor → platform → base  ✅ 允许
base → platform                        ❌ 禁止
editor → workbench                     ❌ 禁止
```

这个约束通过 `build/checker/layersChecker.js` 在 CI 中强制检查。

## 横向切割: 环境目录约定

每个模块内部，按运行环境再细分：

```
vs/platform/files/
├── common/          ← 纯逻辑，可在任何环境运行 (无 DOM、无 Node)
├── browser/         ← 依赖浏览器 API (DOM, Web API)
├── node/            ← 依赖 Node.js API (fs, child_process)
├── electron-main/   ← 仅在 Electron 主进程
├── electron-browser/← Electron 渲染进程 (有 Node 的浏览器环境)
└── test/            ← 测试
```

### 环境标签含义

| 标签 | 可用 API | 运行位置 |
|------|---------|---------|
| `common` | 纯 TypeScript | 任何地方（浏览器、Node、Worker） |
| `browser` | DOM + Web API | Chromium 渲染进程 |
| `node` | Node.js API | 主进程 / Extension Host |
| `electron-main` | Electron Main API | 主进程独占 |
| `electron-browser` | Node + DOM | Electron 渲染进程（有 nodeIntegration） |

### 为什么这样设计？

1. **Web 版复用** — `common` + `browser` 的代码可以直接跑在 vscode.dev (纯浏览器)
2. **编译期隔离** — TypeScript 项目引用保证 browser 代码不会 import node 模块
3. **测试友好** — common 层的逻辑可以在任何环境测试
4. **渐进迁移** — 从 Electron 迁移到 Web 只需替换 node/electron 层实现

## 模块对照: 以文件系统为例

```
vs/platform/files/
├── common/
│   ├── files.ts              ← IFileService 接口定义
│   ├── fileService.ts        ← 通用逻辑 (事件分发、URI解析)
│   └── io.ts                 ← 流式读写抽象
├── browser/
│   ├── htmlFileSystemProvider.ts  ← 浏览器 File System Access API 实现
│   └── indexedDBFileSystemProvider.ts ← IndexedDB 实现
├── node/
│   ├── diskFileSystemProvider.ts  ← Node fs 模块实现
│   └── watcher/                   ← 文件监听 (使用 @parcel/watcher)
└── electron-main/
    └── diskFileSystemProviderServer.ts ← 主进程侧文件服务 (IPC暴露)
```

**同一个接口 `IFileService`，在不同环境有不同实现，通过 DI 容器在运行时绑定正确的实现。**

## platform 层核心模块一览

| 模块 | 职责 | 关键文件/类 |
|------|------|------------|
| `instantiation` | 依赖注入容器 | `ServiceCollection`, `InstantiationService` |
| `files` | 文件系统抽象 | `IFileService`, `DiskFileSystemProvider` |
| `configuration` | 配置管理 (settings.json) | `IConfigurationService` |
| `storage` | 持久化存储 | `IStorageService` (SQLite backed) |
| `ipc` | 进程间通信基础设施 | `IChannel`, `IServerChannel` |
| `extensions` | 扩展元数据与激活 | `IExtensionService` |
| `lifecycle` | 应用生命周期 | `ILifecycleService` |
| `window` | 窗口管理抽象 | `IWindowService` |
| `commands` | 命令注册与执行 | `CommandsRegistry` |
| `contextkey` | 上下文条件表达式 | `ContextKeyExpr` |

## workbench 层结构

```
vs/workbench/
├── api/          ← 插件 API 层 (Extension Host ↔ Main 的桥接)
├── browser/      ← Workbench UI 框架 (布局、Parts)
├── common/       ← 共享类型和工具
├── contrib/      ← 功能贡献点 (每个文件夹=一个IDE功能模块)
│   ├── terminal/
│   ├── debug/
│   ├── scm/
│   ├── search/
│   ├── files/
│   └── ...       ← 50+ 个功能模块
├── services/     ← Workbench 级服务 (比 platform 更上层)
└── electron-browser/  ← Electron 特有逻辑
```

### contrib 的设计理念

`contrib/` 目录下每个子目录就是 IDE 的一个**功能特性**，它们：
- 向 Workbench 注册自己的视图、命令、菜单项
- 通过 `registerSingleton()` 提供服务实现
- 互相之间**不直接依赖**，而是通过 platform/workbench services 间接协作

这就是 VSCode 的**贡献点模式 (Contribution Pattern)**。

## 实际意义: 如何做二次开发

理解了分层，就知道：
1. **定制 UI** → 改 `workbench/browser/` 和 `workbench/contrib/`
2. **加原生能力** → 加 `platform/xxx/node/` 或 `electron-main/`
3. **换品牌** → 改 `product.json` (定制 IDE 的标准做法)
4. **加新语言支持** → 写插件，不改源码
5. **发布 Web 版** → 确保功能在 `common/` + `browser/` 有实现

## 小结

| 设计决策 | 第一性原理 | 收益 |
|---------|-----------|------|
| 纵向四层 | 依赖只能向下 | 编译隔离、独立测试、降低认知负荷 |
| 横向环境标签 | 一套代码多端运行 | Desktop/Web/Remote 统一架构 |
| contrib 贡献点 | 功能正交、松耦合 | 50+ 功能并行开发不冲突 |
| layersChecker | 约束是第一生产力 | CI 自动校验，防止架构腐化 |
