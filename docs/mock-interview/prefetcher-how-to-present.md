# PreFetcher 项目 — 面试表达指南

> 目标：帮你在面试中用 2-3 分钟结构化地讲清楚这个项目，展示技术深度。

---

## 一、开场：30 秒讲清背景（WHY）

面试官问"介绍一下你的项目"时，用这个公式：

**场景 → 痛点 → 方案 → 效果**

> 参考话术：
>
> "这个项目是在美团小程序场景下做的。我们发现页面加载链路中，数据请求是在页面 JS
> 执行完 onLoad 之后才发起的，而页面初始化（WebView 创建、JS 加载）这段时间网络
> 是空闲的。这就导致请求和渲染是串行的，白屏时间长，秒开率只有 X%。
>
> 我设计了一个 PreFetcher 预请求系统，核心思路是在路由跳转的瞬间就提前发起下一个
> 页面的数据请求，利用页面初始化的时间窗口并行加载数据。上线后秒开率提升了 X%。"

**关键点**：
- 要说清楚"为什么慢"（串行链路），不是笼统说"页面慢"
- 要有数据（秒开率从 X% 到 Y%），没有精确数据就说"显著提升"
- 一句话点出核心思路：利用页面初始化的时间窗口

## 二、架构：1 分钟讲清设计（WHAT + HOW）

### 2.1 一句话总结架构

> "整个系统采用**注册-触发-获取**三端分离的设计，本质上是一个发布-订阅模式的变体。"

### 2.2 三端各自职责

| 端 | 职责 | 在哪执行 | 类比 |
|---|---|---|---|
| Register（注册端） | 定义哪些页面需要预请求、请求什么接口 | app.js 启动时 | 订阅 |
| Emit（触发端） | 路由跳转时触发预请求 | 路由拦截层（AOP） | 发布 |
| GetResult（获取端） | 页面 onLoad 时取预请求结果 | 目标页面 | 消费 |

### 2.3 核心数据结构（画图讲）

```
pagePath2InfoMap (Map)
  │
  ├── '/pages/order' → {
  │     taskMap: Map {
  │       'getUserInfo'  → { task: fn, resPromise: Promise }
  │       'getOrderList' → { task: fn, resPromise: Promise }
  │     },
  │     injectParams: { token: 'xxx' },
  │     mergedParams: { token: 'xxx', orderId: '123' }
  │   }
  │
  └── '/pages/detail' → { ... }
```

**面试时画这个图，边画边讲，比纯口述清晰 10 倍。**

### 2.4 为什么这样设计（展示设计思维）

面试官一定会问"为什么三端分离"，回答要点：

1. **关注点分离**：注册关心"请求什么"，触发关心"什么时候请求"，获取关心"怎么用数据"
2. **开闭原则**：新增页面预请求只需要加注册配置，触发和获取逻辑不用改
3. **可测试性**：三端可以独立单测
4. **复用性**：触发端的 AOP 路由拦截是通用的，不依赖具体业务

## 三、技术亮点：展示深度（DEEP DIVE）

面试官追问时，按这个顺序展开：

### 3.1 AOP 路由拦截（设计模式）

**你要表达的核心观点**：
> "预请求的触发是通过 AOP 切面编程实现的，本质是猴子补丁——保存原函数引用，
> 替换为增强函数，在调用原函数之前插入预请求逻辑。这样做到了零侵入接入。"

**跨框架对比**（展示广度）：
- 小程序：劫持 wx.navigateTo（没有路由钩子，只能猴子补丁）
- Vue：router.beforeEach（框架提供了钩子）
- React：history.listen（监听路由变化）

**升华一句**：
> "实现方式不同，但本质都是在路由变化的时机插入逻辑，这就是 AOP 的价值——
> 让增强逻辑和业务逻辑解耦。"

### 3.2 重试机制（容错设计）

**你要表达的核心观点**：
> "预请求失败不能影响用户体验，所以我设计了重试机制，有次数和时长双重保护。
> 如果改进的话，可以用指数退避策略（Exponential Backoff），避免服务端短暂
> 不可用时立即重试造成雪崩。"

**关键术语**：
- 指数退避（Exponential Backoff）：1s → 2s → 4s → 8s
- 抖动（Jitter）：加随机偏移，避免多客户端同时重试
- 降级：重试耗尽后，页面正常发起请求，用户无感知

### 3.3 竞态与去重（并发控制）

**你要表达的核心观点**：
> "预请求和页面请求共享同一个 Promise 引用（resPromise），天然去重。
> 连续跳转同一页面时，通过手动维护 status 字段判断是否有 pending 的请求，
> 避免重复发起。"

**为什么要手动维护 status**：
> "原生 Promise 不暴露内部状态（pending/fulfilled/rejected），
> 所以需要在业务层做状态代理。"

### 3.4 内存管理（生产意识）

**你要表达的核心观点**：
> "预请求结果存在内存 Map 里，如果用户没有真正到达目标页面，数据会残留。
> 解决方案：
> 1. 页面 onLoad 消费后清除对应的 resPromise
> 2. 设置 TTL 过期时间，超时自动清理
> 3. 监听路由回退事件，主动清理未消费的预请求"

## 四、版本演进：展示成长（GROWTH）

面试官喜欢听"你是怎么一步步做到的"，而不是"一上来就完美"。

> "这个系统经历了三个版本：
>
> V1 我先实现了最小可用版本，就是基本的注册-触发-获取流程，验证了预请求
> 方案的可行性，数据上确实能缩短白屏时间。
>
> V2 引入了 TypeScript，把 taskQueue 改成了 taskMap，支持按 key 获取
> 特定任务的结果，灵活性更好。
>
> V3 加入了重试机制和完善的错误处理，让系统在生产环境更可靠。"

**这个叙事展示了**：MVP 思维 → 工程化改进 → 生产级完善

## 五、扩展思考：展示前瞻性

面试官问"还能怎么优化"时：

1. **预测式预请求**：根据用户行为数据预测下一步操作，提前预请求（类似浏览器的 prefetch）
2. **请求取消**：用 AbortController 在路由回退时取消未完成的预请求
3. **缓存整合**：预请求结果写入缓存层，页面优先读缓存，实现多级加速
4. **优先级调度**：高优先级任务先执行，低优先级任务在空闲时执行（类似 React Scheduler）
