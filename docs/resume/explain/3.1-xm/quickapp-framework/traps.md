## 雷 1：你在框架里的角色

**可能的问法**："框架是你设计的还是你接手的？你具体负责什么？"

**回答**：

框架是团队多年积累的产品，我加入后负责**性能治理和工程化演进**：
- 包体 153MB → 60MB（模块裁剪方案设计）
- 启动内存 PSS 41MB → 35.8MB（DEX 布局优化）
- 自动化测试体系搭建（三端设备自动化）
- 条件编译工具套件（开源）

做这些事的前提是**深入理解框架整体架构**（三线程模型/J2V8 Bridge/渲染引擎），才能知道瓶颈在哪、该从哪一层优化。

**关键表达**：
- 不说"我从零设计的"（诚实）
- 强调"做优化必须深入理解架构"（证明有框架级认知）
- 把"优化"讲成"治理/演进"（架构师视角，不是执行者）

## 雷 2：J2V8 同步 Bridge 实现细节

**可能的问法**：
- J2V8 怎么注册一个 Java 方法给 JS 调用？
- 同步调用时线程怎么处理？会不会阻塞 UI 线程？
- 和 JSI 的实现层面区别？

### 本质

V8 只暴露 C++ API → Java 不能直接调 → J2V8 通过 JNI 把 V8 的 C++ API 封装成 Java API。本质就是一层 JNI 封装。

### 怎么注册方法

```
1. 创建 V8Object（JS 侧的桥接对象）
2. registerJavaMethod(实例, Java方法名, JS方法名, 参数类型)
   → 把 Java 方法绑定到 V8 的 Host Object 上
3. runtime.add("NativeBLE", bridge)
   → 挂到 JS 全局

效果：JS 调 NativeBLE.connect(id) → V8 通过 JNI 回调 Java 的 connect 方法 → 同步返回
```

### 线程处理

**核心规则**：V8 Runtime 绑定创建它的线程（JS Thread）。

- JS 调 Java → 在 JS Thread 上同步执行（不在 UI Thread）
- **不会直接阻塞 UI Thread**
- 但如果 Java 方法做了耗时操作 → 阻塞 JS Thread → JS 无法产生渲染指令 → 间接影响 UI
- 最佳实践：快速操作同步返回，耗时操作开子线程 + 回调

### 和 JSI 的区别

| 维度 | J2V8 | JSI |
|------|------|-----|
| 绑定层 | Java 直接绑定 V8 | C++ 抽象接口 |
| 引擎 | 绑定 V8 | 引擎无关（Hermes/V8/JSC） |
| 性能 | 略慢（多一层 JNI） | 更快（纯 C++） |
| 开发效率 | 高（Java 生态） | 低（需写 C++） |

**本质相同**：都是"让 JS 能同步调用宿主方法"，去掉异步中间层。

## 雷 3：DEX 布局优化具体怎么做

**可能的问法**：
- 热代码列表怎么收集的？
- 怎么让 R8/D8 按指定顺序排列类？
- page fault 减少了多少？怎么测量的？

### 本质问题

App 启动时 ART 从 DEX 文件加载类。如果启动路径用到的类（热代码）分散在 DEX 各处 → 大量随机磁盘读取（page fault）→ 启动慢 + 内存高。

**解决**：把热代码集中排列在 DEX 前部 → 顺序读取 → 减少 page fault。

### 热代码列表怎么收集

三种方式：

| 方式 | 做法 | 适用 |
|------|------|------|
| Baseline Profile（官方） | Macrobenchmark 跑启动路径，自动生成 profile | 标准 App |
| 线上采集 | ClassLoader hook 记录启动加载的类，上报聚合 | 大厂精细优化 |
| 手动标注 | 开发者列出核心类列表 | 快速验证 |

实际做法：Baseline Profile 为主 + 手动补充。

### 怎么让 R8/D8 按顺序排列

- 生成 `baseline-prof.txt`（或 `art-profile.txt`），列出热代码类和方法
- 打包时 R8/D8 读取这个文件，把这些类优先排列在 primary DEX 前部
- Gradle 配置：`baselineProfile` 插件自动处理

### 怎么测量效果

- **page fault**：`adb shell dumpsys meminfo <package>` 看 "Page Faults" 字段
- **PSS**：`adb shell dumpsys meminfo <package>` 看 PSS 值
- **启动时间**：`adb shell am start -W` 看 TotalTime
- 优化前后对比：PSS MAX 41MB → 35.8MB

### 为什么 page fault 减少 = 内存降低

page fault 触发时系统把磁盘页加载到内存 → 占用物理内存（PSS）。page fault 越多 → 加载的页越多 → PSS 越高。热代码集中后，同样的启动路径只需要加载更少的页 → PSS 降低。

## 雷 4：反射解耦的性能开销

**可能的问法**：
- 反射比直接调用慢多少？
- 高频调用的模块也用反射吗？
- 有没有缓存 Method/Class 引用？

### 本质

反射 = 运行时通过字符串查找类/方法，而不是编译时直接链接。查找有开销，但可以缓存。

### 性能数据

| 操作 | 耗时 |
|------|------|
| 直接方法调用 | ~1-5 ns |
| 反射调用（首次，含查找） | ~1000-5000 ns |
| 反射调用（缓存 Method 后） | ~50-100 ns |

**首次慢 1000x，缓存后只慢 10-20x。**

### 怎么处理

1. **只在模块初始化时用反射**（低频，一次性）→ 开销可忽略
2. **缓存 Class/Method 引用**：首次反射查找后缓存，后续直接用缓存的引用调用
3. **高频调用不用反射**：模块初始化后拿到实例，后续直接调实例方法（不再反射）

```
// 伪代码
// 初始化时（一次性，反射）
Class<?> moduleClass = Class.forName("com.xxx.MapModule");
Object instance = moduleClass.newInstance();
Method initMethod = moduleClass.getMethod("init");
initMethod.invoke(instance);  // 反射调用

// 后续使用（高频，直接调用）
IModule module = (IModule) instance;
module.bindDevice(deviceId);  // 接口调用，不是反射
```

### 为什么不用注解处理器（APT）替代反射

| 方案 | 优点 | 缺点 |
|------|------|------|
| 反射 | 简单、运行时灵活、不增加编译复杂度 | 首次有性能开销 |
| APT（编译时生成代码） | 零运行时开销 | 增加编译复杂度、生成代码难调试 |

对于"模块初始化"这种低频场景，反射的开销可以忽略，不值得引入 APT 的复杂度。

## 雷 5：为什么不直接用 RN

**可能的问法**：
- 快应用是系统级预装，为什么不能用 RN？
- 自研框架的优势/劣势？
- 如果重新选，还会自研吗？

### 为什么不用 RN

| 约束 | RN 的问题 | 自研的解决 |
|------|----------|-----------|
| **包体** | RN 框架本身 ~15-20MB（Hermes + 框架代码） | 自研可以极致裁剪，只保留需要的能力 |
| **预装 ROM** | RN 版本升级需要跟随 App 发版 | 自研框架和系统绑定，可以随 OTA 升级 |
| **启动速度** | RN 启动有固定开销（框架初始化） | 自研可以针对系统级场景极致优化启动路径 |
| **定制深度** | RN 是通用框架，改底层成本高 | 自研完全可控，想改哪改哪 |
| **协议/标准** | 快应用有自己的标准（W3C 快应用联盟） | 需要实现特定标准，RN 不直接支持 |

### 自研的劣势

- 生态小（没有 RN 的海量第三方库）
- 维护成本高（所有 bug 自己修）
- 招人难（会 RN 的人多，会快应用框架的人少）

### 如果重新选？

> 如果是做一个普通 App → 用 RN，不自研。
> 如果是做系统级预装运行时（包体极致敏感 + 需要和 ROM 深度集成 + 有自己的标准）→ 仍然需要自研。
> 这不是"RN 好不好"的问题，是"场景约束决定技术选型"。
