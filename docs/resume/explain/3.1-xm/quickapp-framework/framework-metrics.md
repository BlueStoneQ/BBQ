## 包体指标

### 为什么重要

- 预装在系统 ROM 中，每个用户都占空间
- OTA 升级时包体大 = 下载慢 = 升级率低
- 低端机存储有限（4GB/8GB 设备）

### 治理方案

| 手段 | 原理 | 效果 |
|------|------|------|
| **R8 混淆 + 优化** | 代码缩减、类合并、方法内联 | DEX -15~25% |
| **条件编译** | 按目标平台裁剪无用代码（AOT 编译期） | 去除不需要的 Feature 代码 |
| **模块裁剪 + 反射解耦** | 非核心模块不编译进主包，运行时按需加载 | 预装包 -60% |
| **资源压缩** | PNG → WebP、移除未引用资源 | 资源 -30~50% |
| **ProGuard/R8 规则优化** | 精细化 keep 规则，避免过度保留 | DEX 进一步缩减 |
| **SO 库裁剪** | 只保留目标 ABI（arm64-v8a），strip debug symbols | SO -40~60% |

### 监控方式

```
CI 构建后自动检测包体：
  - 总包大小 vs 基线（超过阈值告警）
  - DEX 大小趋势图
  - 新增依赖自动检测（引入新 SDK 需审批）
```

## 启动指标

### 为什么重要

- 快应用定位"秒开"，启动慢就失去了存在意义
- 系统预加载运行时，但页面加载仍有开销
- 低端机内存紧张，启动时 GC 频繁会卡顿

### 启动链路

```
用户点击图标
  → 系统 fork 进程（或复用预热进程）
  → 加载 DEX（类加载）
  → 初始化框架运行时（V8 引擎、J2V8 环境）
  → 解压/读取 rpk
  → 执行 app.js + 页面 JS
  → JS 生成 Virtual DOM
  → J2V8 调用 Native 创建 View
  → 首帧渲染（用户可见）
```

### 治理方案

| 手段 | 优化哪个环节 | 效果 |
|------|------------|------|
| **DEX 布局优化** | 类加载（减少 page fault） | PSS MAX -12%，冷启动 -200ms |
| **V8 Snapshot** | V8 初始化（预编译 JS 上下文） | JS 引擎启动 -100ms |
| **预热进程** | fork 进程 | 用户点击时进程已就绪 |
| **rpk 预解压** | 解压 rpk | 首次打开 -150ms |
| **JS 预编译（字节码缓存）** | 执行 JS | 二次启动 JS 解析 -50% |
| **懒加载非首屏模块** | 页面 JS 执行 | 首屏只加载必要代码 |
| **Native View 复用池** | 创建 View | 页面切换时复用已有 View |

### 监控方式

```
自动化测试（pytest + uiautomator2）：
  - 冷启动：kill 进程 → 点击 → 首帧时间
  - 热启动：后台切回 → 首帧时间
  - 多机型覆盖（高/中/低端）
  - 每日跑，趋势告警
```

## 运行时指标

### 为什么重要

- 帧率低 = 用户感知卡顿
- 通信耗时高 = 交互延迟
- 内存泄漏 = 长时间使用后 OOM

### 治理方案

| 指标 | 常见瓶颈 | 治理手段 |
|------|---------|---------|
| **帧率（FPS）** | JS 执行阻塞、过度布局、大图渲染 | 长任务拆分、布局层级优化、图片降采样 |
| **通信耗时** | 旧 Bridge 异步序列化 | J2V8 同步调用（已优化到微秒级） |
| **内存泄漏** | V8 和 Java 侧交叉引用未释放 | Release 回调、弱引用、定期 GC |
| **页面切换** | 新页面 JS 解析 + View 创建 | 预加载、View 复用池、字节码缓存 |

### 监控方式

```
线上：
  - FPS 采集（Choreographer 回调计算丢帧）
  - 内存水位监控（PSS 定时采样）
  - JS 执行耗时埋点（超过 16ms 的任务上报）

线下：
  - Systrace / Perfetto 分析帧耗时
  - LeakCanary 检测 Java 侧泄漏
  - V8 heap snapshot 检测 JS 侧泄漏
```

## 稳定性指标

### 为什么重要

- 框架 Crash = 所有快应用不可用（影响面极大）
- ANR = 系统弹窗"应用无响应"（用户直接杀进程）
- 兼容性差 = 三方开发者流失

### 常见 Crash 原因

| 原因 | 场景 | 防治 |
|------|------|------|
| **JNI 空指针** | V8 GC 后 Java 侧仍持有已释放对象 | Release 回调 + weak reference |
| **OOM** | 低端机内存不足 | 内存水位监控 + 主动释放缓存 |
| **Native Crash（SO）** | V8 引擎 bug、JNI 类型错误 | Breakpad 捕获 + 符号化 |
| **JS 异常未捕获** | 三方快应用代码错误 | try-catch 兜底 + 错误上报 |
| **线程死锁** | JS 线程和 UI 线程互等 | 超时机制 + 线程模型规范 |

### 常见 ANR 原因

| 原因 | 场景 | 防治 |
|------|------|------|
| **主线程 IO** | 读取大文件、同步网络请求 | StrictMode 检测 + 异步化 |
| **JS 长任务阻塞** | 复杂计算占满 JS 线程 | 任务拆分 + Worker |
| **锁竞争** | 多线程访问共享资源 | 减少锁粒度、无锁数据结构 |

### 监控方式

```
线上：
  - Crash 上报（Bugly / Firebase Crashlytics）
  - ANR 监控（Watchdog 线程检测主线程心跳）
  - 兼容性：线上快应用启动成功率

线下：
  - 自动化 Monkey 测试（随机操作 + Crash 检测）
  - 压力测试（连续启动/退出 100 次）
  - 内存压力测试（模拟低内存场景）
```

## 治理方案

### 整体治理框架

```
指标采集 → 基线建立 → 劣化检测 → 归因分析 → 优化实施 → 效果验证
    │           │           │           │           │           │
    ▼           ▼           ▼           ▼           ▼           ▼
  埋点/工具   历史数据    CI 卡点     Profiler    代码改造    A/B 对比
```

### 防劣化机制

| 环节 | 手段 |
|------|------|
| **开发时** | 性能 Lint 规则（禁止主线程 IO、限制布局层级） |
| **CI** | 包体阈值卡点、启动时间基准测试 |
| **发布前** | 自动化性能测试（多机型冷启动/帧率/内存） |
| **发布后** | 线上监控 + 趋势告警（指标连续 3 天劣化触发） |

## 常见问题

### Q: 框架指标和业务 App 指标有什么区别？

> 框架关注的是"底座能力"——启动速度、通信效率、内存占用、稳定性。业务 App 关注的是"用户体验"——首屏时间、交互响应、业务成功率。框架指标是业务指标的下限——框架慢，业务不可能快。

### Q: 怎么平衡包体和功能？

> 模块化 + 按需加载。核心模块（渲染引擎、通信层）必须预装，扩展模块（地图、支付、蓝牙）按需下载或反射加载。用条件编译在编译期裁剪，用反射解耦在运行时按需加载。

### Q: 低端机怎么保证体验？

> 分级策略：检测设备等级（RAM/CPU），低端机关闭动画、降低渲染精度、限制并发任务数、更积极地释放缓存。框架层做好降级，业务层无感知。
