## 一、分发方式对比

| 方式 | 签名 | 更新方式 | 包体要求 | 卸载 |
|------|------|---------|---------|------|
| 商店发布 | 开发者 keystore | 商店推送 | 相对宽松 | 用户可卸载 |
| ROM 预装 | 系统 platform key | OTA / 自升级 | 极严格 | 不可卸载 |
| 企业分发 | 企业证书 | MDM 推送 | 无限制 | 管理员控制 |
| 侧载（adb install） | 任意 | 手动 | 无限制 | 可卸载 |

## 二、商店上架流程

### Google Play 流程

```
1. 注册开发者账号（$25 一次性）
2. 创建应用 → 填写商店信息（描述/截图/分类）
3. 上传 AAB（Android App Bundle）
4. 设置签名（App Signing by Google Play）
5. 提交审核（通常 1-3 天）
6. 发布（全量 / 分阶段灰度）
```

### 国内商店（小米/华为/OPPO/vivo）

- 每个商店独立提交（没有统一入口）
- 需要软件著作权证书
- 审核标准各不相同
- 通常用 APK 而不是 AAB

### AAB vs APK

| | APK | AAB (Android App Bundle) |
|---|---|---|
| 格式 | 完整安装包 | 模块化包（Google Play 按设备生成 APK） |
| 大小 | 包含所有 ABI/语言/密度资源 | 按需分发（用户只下载自己设备需要的） |
| 签名 | 开发者签名 | Google Play 重新签名 |
| 适用 | 国内商店 / 侧载 / ROM 预装 | Google Play |

## 三、签名机制

### 签名的本质

APK 签名 = 用私钥对 APK 内容做数字签名，证明：
1. **身份**：这个 APK 是我发布的
2. **完整性**：APK 没有被篡改

### Keystore

```
keystore 文件（.jks / .keystore）
├── 私钥（签名用，绝对不能泄露）
├── 公钥证书（验证用，打包进 APK）
└── 别名（alias，一个 keystore 可以有多个密钥对）
```

### 签名方案演进

| 版本 | 方案 | 特点 |
|------|------|------|
| v1 | JAR 签名 | 逐文件签名，慢，可以修改 META-INF |
| v2 | APK Signature Scheme v2 | 整个 APK 签名，快，不可修改任何字节 |
| v3 | 支持密钥轮换 | 可以更换签名密钥（v2 不行） |
| v4 | 增量安装优化 | ADB 增量安装加速 |

### 系统签名（Platform Key）

ROM 预装应用用系统签名：
- 拥有 `android:sharedUserId="android.uid.system"` 权限
- 可以访问系统级 API
- 不可被普通 APK 覆盖安装（签名不匹配）

## 四、ROM 预装流程

```
框架代码开发完成
  ↓
CI 构建 → 产出 APK（用系统 platform key 签名）
  ↓
提交到 ROM 代码仓库（通常是 Gerrit）
  ↓
ROM 构建系统（通常是 Android.mk / Android.bp）
  ↓ 编译整个 ROM
产出 ROM 镜像（system.img / vendor.img）
  ↓
刷机验证（fastboot / MiFlash）
  ↓
OTA 包生成 → 推送给用户
```

### 预装位置

```
/system/app/          ← 普通系统应用（可被用户"停用"但不能卸载）
/system/priv-app/     ← 特权系统应用（拥有更多权限）
/product/app/         ← 产品定制应用
/vendor/app/          ← 厂商定制应用
```

### 预装应用的约束

- 包体影响 ROM 大小（ROM 有总大小限制）
- 不能随意更新（需要等 OTA 或走自升级）
- 稳定性要求极高（crash 影响系统评价）
- 内存有合规上限（PSS MAX）

## 五、OTA 与自升级

### OTA（Over The Air）

系统级更新，更新整个 ROM 分区：

```
服务器推送 OTA 包
  ↓ 用户确认
下载差分包（只包含变化的部分）
  ↓
A/B 分区切换（无缝更新，不影响使用）
  ↓
重启后生效
```

OTA 周期长（通常月级），不适合频繁更新。

### 自升级

预装应用自己检查更新，下载新版本覆盖安装：

```
App 启动 → 检查服务器版本
  → 有新版本 → 下载 APK
  → 静默安装（系统应用有权限静默安装）
  → 下次启动生效
```

快应用框架的策略：
- 预装包是精简版（buildForRom=true，砍掉凡泰/百度）
- 用户首次使用时触发自升级 → 下载完整版覆盖
- 这样既满足 ROM 包体要求，又不影响功能完整性

## 六、快应用框架的分发策略

```
预装版（ROM 集成）
  - buildForRom=true
  - 砍掉非核心模块（凡泰/百度）
  - 包体 ~60MB
  - 系统签名
  - 满足 PSS 合规

完整版（自升级）
  - buildForRom=false
  - 包含所有模块
  - 包体 ~153MB
  - 开发者签名（或系统签名）
  - 用户首次使用时自动升级
```

这个"预装精简 + 自升级恢复"的策略，就是你在简历里写的"模块裁剪与降级方案：反射解耦编译依赖 + metadata 入口控制 + 自升级兜底"。
