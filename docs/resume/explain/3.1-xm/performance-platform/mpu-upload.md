## 背景

线上最大数据包 11.4G，原方案（业务服务器直写 JuiceFS 挂载目录）在 5G+ 时失败，且大文件上传耗时过长。

## 方案选型

| 方案 | 说明 | 问题 |
|------|------|------|
| 业务服务器直写（原方案） | 前端流式上传到 BFF，BFF 写入 JuiceFS | 大文件慢，11.4G 失败 |
| 自定义分片上传 | 前端切片 → 逐片上传到业务服务器 → 服务端合并 | IO 处理耗时高，比整传还慢 |
| S3 整传 | 前端直传 S3（预签名 URL） | 单次上传限制 5G |
| **S3 MPU 分片并发上传（选）** | 前端切片 → 并发 PUT 到 S3 → 合并 | 无大小限制，速度最快 |

## 核心数据

| 文件大小 | 原方案（直写） | S3 MPU 并发 | 提速 |
|---------|-------------|------------|------|
| 1.3G | 30.8s | 12.7s | ~2.4x |
| 2G | 41.8s | 17.9s | ~2.3x |
| 4.4G | 121.4s | 42.5s | ~2.9x |
| 5.7G | 155.8s | 51.7s | ~3x |
| 11.4G | 失败 | 100.1s | 突破上限 |

## 实现要点

- 分片大小：64MB（2 的幂次对齐）
- 并发数：6（S3 支持乱序并发上传）
- 流程：后端一次请求返回所有 URL（init + presign + completeUrl）→ 前端并发 PUT → 前端直接 POST completeUrl 合并
- 按文件大小分流：< 4GB 走原有单次 PUT，≥ 4GB 走 MPU

## 为什么并发数是 6？

**本质**：浏览器对同一 origin 的 HTTP/1.1 并发 TCP 连接数上限为 **6**（Chrome 默认值）。

- 超过 6 个请求会排队等待空闲连接，不会更快
- 低于 6 个浪费了可用连接，带宽利用不满
- 所以 6 是**打满浏览器连接池的最优并发数**

注意：
- 这不是"6 个线程"——浏览器网络层是事件驱动的，不是多线程发请求
- 如果 S3 endpoint 支持 HTTP/2 多路复用，理论上单连接可并发更多流，但 presigned URL 场景实测 6 已经瓶颈在带宽而非连接数
- 不同浏览器限制不同（Firefox 也是 6，Safari 是 6），6 是安全的通用值

**面试话术**：

> "并发 6 是因为浏览器对同一域名的 HTTP/1.1 并发 TCP 连接上限是 6。超过 6 会排队，低于 6 浪费带宽。实测瓶颈已经在网络带宽而非连接数，所以 6 是最优值。"

## 断点续传

### 核心思路

**记录已上传分片的状态，中断恢复后跳过已完成的分片继续上传。**

### 实现方案

```
① CreateMultipartUpload → 拿到 uploadId
   → 持久化到 localStorage: { fileHash, uploadId, totalParts }

② 每个分片 PUT 成功
   → 记录到 localStorage: upload_{fileHash}_parts = [{ partNumber, etag }, ...]

③ 中断恢复时（刷新/断网重连）：
   - 读取本地 uploadId → 不需要重新 CreateMultipartUpload
   - 对比已上传分片列表 → 只传缺失的分片
   - （可选）调 S3 ListParts API 验证服务端状态，防止本地记录不一致

④ 全部分片完成 → CompleteMultipartUpload

⑤ 清理本地状态
```

### 文件标识（fileHash）

- 用 `文件名 + 文件大小 + lastModified` 生成唯一标识
- 或对文件头部（前几 MB）做 MD5/SHA-256（大文件全量 hash 太慢）
- 目的：同一文件重新上传时能匹配到之前的进度

### 关键细节

| 问题 | 解决 |
|------|------|
| uploadId 过期（S3 默认 7 天） | 本地记录带时间戳，超期则重新 Create |
| 本地记录和服务端不一致 | 恢复时调 ListParts 校验，以服务端为准 |
| 用户换浏览器/清缓存 | 降级为重新上传（可接受，非核心路径） |
| 并发上传中某个分片失败 | 单片重试 3 次，仍失败则标记跳过，最终提示用户 |

### 面试话术（30秒版）

> "断点续传的核心是持久化上传进度。我用 localStorage 记录 uploadId 和每个分片的完成状态（partNumber + etag）。页面刷新或网络恢复后，读取本地记录，跳过已完成的分片，只上传剩余部分。S3 MPU 天然支持这个——uploadId 在服务端保持有效（7天），分片可以分多次上传，最后统一 Complete。恢复时还会调 ListParts 和服务端校验，防止本地记录不一致。"

## 为什么自定义分片方案反而更慢？

耗时分析（2G 文件，总耗时 129s）：
- 服务端 IO 处理：70.8s（占 55%）— 频繁切片文件读写
- 切片网络请求：34.5s（占 27%）— 串行逐片上传
- 合并耗时：22.5s（占 17%）

根因：业务服务器做中转，每个切片要写临时文件再合并，IO 开销巨大。S3 MPU 直接传到对象存储，不经过业务服务器，省掉了中转 IO。
