# CRN 企业级定制

> 第一性原理：标准 RN 满足不了大型 App 的工程化需求，怎么办？

---

## 一、CRN 是什么

CRN = 某大厂基于 React Native 的企业级定制框架。本质是在标准 RN 之上加了一层**工程化增强**，解决大型 App 使用 RN 时的实际痛点。

类比：RN 是毛坯房，CRN 是精装修 + 物业管理。

---

## 二、标准 RN 在大型 App 中的问题

| 问题 | 场景 | 影响 |
|------|------|------|
| 单 Bundle 过大 | 所有业务打成一个 JS Bundle | 首屏加载慢、内存占用高 |
| 无法独立发版 | 业务 A 改了代码，业务 B 也要跟着发版 | 发版效率低、风险高 |
| 热更新粗粒度 | 整个 Bundle 替换 | 更新包大、下载慢 |
| 缺少性能监控 | 不知道哪个页面慢、哪里内存泄漏 | 线上问题难排查 |
| 多团队协作难 | 代码耦合、构建冲突 | 开发效率低 |

---

## 三、CRN 的核心解决方案

### 3.1 分 Bundle（拆包）

**问题**：一个 App 有几十个 RN 页面，全打成一个 Bundle 太大。

**方案**：
```
┌─────────────────────────────────────┐
│  Common Bundle（公共依赖）            │
│  react / react-native / 公共组件     │
├─────────────────────────────────────┤
│  Business Bundle A（机票频道）        │
│  Business Bundle B（酒店频道）        │
│  Business Bundle C（火车票频道）      │
└─────────────────────────────────────┘
```

- **Common Bundle**：包含 RN 框架代码和公共依赖，App 启动时预加载
- **Business Bundle**：每个业务频道独立打包，按需加载
- 好处：首屏只加载 Common + 当前业务，其他业务不影响

**技术实现**：
- 构建时：Webpack/Metro 配置 entry 拆分，DLL 提取公共依赖
- 运行时：自定义 Bundle Loader，按路由动态加载对应 Business Bundle
- 版本管理：Common 和 Business 独立版本号，独立灰度

### 3.2 热更新（动态发版）

**问题**：App Store 审核慢（iOS 1-7 天），紧急 bug 修不了。

**方案**：
```
App 启动 → 检查更新 → 下载差量包 → 合并 → 下次启动生效
```

- **差量更新**：不下载整个 Bundle，只下载 diff（bsdiff 算法）
- **分版本管理**：不同 App 版本对应不同 Bundle 版本
- **灰度发布**：按用户 ID/地区/比例灰度下发
- **回滚机制**：检测到 crash 率上升自动回滚

**与 CodePush 的区别**：
| 维度 | CodePush | CRN 热更新 |
|------|----------|-----------|
| 粒度 | 整个 Bundle | 分 Bundle 独立更新 |
| 差量 | 有 | 有，更精细 |
| 灰度 | 基础 | 企业级（多维度） |
| 监控 | 基础 | 集成性能监控 |
| 回滚 | 手动 | 自动（crash 率触发） |

### 3.3 容器化

**问题**：RN 页面和 Native 页面混合，生命周期管理复杂。

**方案**：
- RN 容器（RNContainer）：封装 RN 实例的创建/销毁/复用
- 预加载：App 启动时预创建 RN 容器，打开 RN 页面时直接复用（秒开）
- 多实例管理：支持同时存在多个 RN 实例，互不干扰
- 内存回收：后台页面自动释放，前台恢复

### 3.4 性能监控

- 页面加载耗时（分阶段：Bundle 加载 → JS 执行 → 首屏渲染）
- FPS 监控（掉帧检测）
- 内存监控（泄漏检测）
- Crash 上报 + 堆栈还原
- 自定义埋点（业务指标）

## 四、工程化实践

### 4.1 开发流程

```
本地开发（Hot Reload）
  → 提交代码（Git）
  → CI 构建（分 Bundle 打包）
  → 自动化测试
  → 灰度发布（热更新平台）
  → 全量发布
  → 监控告警
```

### 4.2 多团队协作

- 每个业务团队独立仓库/独立分支
- Common Bundle 由框架团队维护
- Business Bundle 由业务团队独立发版
- 版本兼容性检查（Common 升级时自动检测 Business 兼容性）

---

## 五、与标准 RN 的关系

```
标准 RN（Meta 开源）
  │
  ├── 渲染引擎（不改）
  ├── JS 引擎（可能换 Hermes → 自研）
  ├── Bridge/JSI（不改）
  │
  └── CRN 增强层（企业级定制）
       ├── 分 Bundle 方案
       ├── 热更新系统
       ├── 容器化管理
       ├── 性能监控
       └── 工程化工具链
```

**CRN 不改 RN 底层**，而是在上层做工程化增强。这和快应用框架不同——快应用是从底层重新实现。

## 六、关键概念速查

| 概念 | 一句话解释 |
|------|-----------|
| 分 Bundle | 把一个大 JS Bundle 拆成公共包 + 多个业务包，按需加载 |
| 热更新 | 不经过 App Store，直接下发 JS Bundle 更新到用户设备 |
| 差量更新 | 只下载变化的部分（diff），不下载整个 Bundle |
| 容器化 | 封装 RN 实例的生命周期管理，支持预加载和复用 |
| 灰度发布 | 按比例/条件逐步放量，而不是一次性全量更新 |
| Common Bundle | 公共依赖包（RN 框架 + 公共组件），所有业务共享 |
| Business Bundle | 业务独立包，每个频道/模块独立打包和发版 |
